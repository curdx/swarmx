# 聊天窗口设计方向：「队长在场 / 机器下沉」

> 押注：用户来 swarmx 的核心心智不是「管理一群 agent」，而是「跟一个能干活的队长把事说清楚」。聊天窗口因此首先是一个安静、可信、单声道的对话；多成员的机器细节默认不在场，只在它**真的发生**时，以一张克制的系统卡入流。

---

## 1. 一句话主张

**押注「单声道队长对话」**：屏幕中央永远只有一条人能读完的叙事轴——你的话、队长的话、以及少数几种「真的发生了」的系统卡；成员的 token 流、工具调用、终端、diff 全部默认折叠进右侧抽屉，只在对话流里留下一行诚实的进度。我们用「会话栏 + 对话主轴」两栏吃满屏幕，把现状的「四栏 chrome」和「三层 header」整个删掉，赌大多数时候用户根本不需要看 agent 仪表盘。

---

## 2. 整屏布局

设计原则：**最多两栏常驻**。现状的图标导航栏、工作空间/方向/源码三列、成员栏、三层 header——全部砍掉或下沉。对话是主角，必须占屏幕主体（直接回应现状 P1「四栏 chrome 把对话压成窄带」）。

```

│ ●退款流程 │                                                            │

```

- **对话主轴居中限宽**（max-width ≈ 720px，左右留白），即使只有两条短消息也不会像现状那样「贴顶 + 大片空白显得被抛弃」（回应现状 P1）。
- **状态行只有一条 36px**：状态点 + 会话名 + 五态文字 + 三标签（对话/终端/变更）+ 抽屉开关。删掉现状的 tab 条、breadcrumb 卡、默认视图选择器三层堆叠。

```
┌──────────┬─────────────────────────────────────┬──────────────────────┐

│          │           （对话主轴，居中限宽）       │  ◐队长   工作中       │

│          │   你：把校验抽成独立函数              │  ●测试   写测试 14s   │ ← 成员状态板
│ 退款流程  │   队长：好，我拆两步…                 │  ⚠后端   等你 · 登录   │   (色+态+一行动作)

│          │                                     │  当前 tab：变更         │
│          │                                     │  refund.ts  +12 −3    │ ← 抽屉跟随上方标签

└──────────┴─────────────────────────────────────┴──────────────────────┘
```

- 抽屉**默认收起**，即便 1536px+ 也由用户决定是否常驻（记忆每会话偏好）。它不是现状那种「为 1 个 agent 占一整列、信息几乎不变」的成员栏（回应现状 P2）——只有打开终端/变更/成员细节时才出现。
- 抽屉内是上方三标签（对话/终端/变更）+ 成员状态板的统一容器：**右侧是高频机器输出的家**，左侧对话永远干净。这正是竞品研究里「左叙事 / 右工作面」的主流分屏，只是右面默认折叠。
- `<1024px`：会话栏变 Sheet（汉堡唤出），抽屉变全屏覆盖层，对话独占。

## 3. 消息流渲染

四种渲染物，**只有前两种是气泡**，后两种是系统卡/进度行——这是控制 firehose 的关键。

| 类型 | 形态 | 归属 | 折叠规则 |
|---|---|---|---|
| **你的消息** | 右对齐蓝气泡（`--color-accent-primary` 填充 + 右下尾角），无头像 | — | 永不折叠 |
| **队长消息** | 左对齐，**强对比正文**（深色文字、白底、左侧 26px 蓝头像「队」），名字+时间在气泡上方一行 | 始终 `from=队长` | 默认展开；`思考摘要`默认折叠成一行 |
| **成员消息** | **默认不进流**。成员只通过队长说话 | 抽屉里按色归属 | 仅「交付」「需要你」两种结果才升格为系统卡入流 |
| **系统事件卡** | 居中或带左色条的卡，**形状与气泡明显不同**（派工/交付/计划/失败/模型切换/回退/隔离降级） | 锚定相关成员的角色色 | 派工卡 `▾` 折叠；失败卡永不折叠 |

**关键决策：成员不在流里发气泡**（竞品研究三家一致：Anthropic 多 agent 研究系、Claude Code sub-agents——子 agent 在隔离上下文里干活，只有**凝练结果**冒泡给人）。落到 swarmx：

- 队长是唯一对人说话的「声音」。worker 的 raw token 流、worker→队长 的 handoff，默认是内部协调，**不广播**。
- 现状的「pending responder 气泡」「多 agent merge 推断 reasoning」全部下沉——它们制造的正是 firehose。
- 成员的工作只在三个时刻浮上来：① 队长引用它（「测试成员补好了用例」）；② 它**交付**（→ 交付卡）；③ 它**卡住要你**（→ 需要你卡）。

**修复现状的弱层级问题**（P1「队长气泡几乎隐形」）：队长正文用 `--color-foreground-primary` 深色 + 白底，是全屏最强元素；你的蓝气泡降为次强。AI 输出是 payload，必须最显眼。

**多 agent 并行不变 firehose 的三道闸**：
1. **单声道**：5 个 worker 同时跑，对话流里依然只有队长一条叙事 + 至多 5 张派工卡（折叠态各一行）。
2. **派工卡聚合**：每张派工卡是一个折叠父行，展开才显示该成员的子步骤/工具调用（竞品研究的「两级折叠的 worker 行」模式，对应现状 `agentActivityById`）。
3. **进度行不是气泡**：worker 活着的信号是**一行小字 + 三点 + 心跳数字**（下一节），不占气泡高度，N 个并行也只是 N 行小字。

**稳定身份**（竞品研究：N>3 时归属是承重设计）：每个成员用**角色派生名 + 固定色**（复用 `--color-agent-*`：planner 蓝 / test 绿 / backend 紫 / frontend 青 / architect 红…），同一个色在派工卡、进度行、抽屉成员板、diff 归属处复用。禁止 `worker_7` 这种裸 ID 泄漏。

**时间分隔 / 未读**：保留现状的 >5min 时间分隔线和「N 条新消息」分隔（`MessagesPanel.tsx:1571-1601`），但未读计数改用精确端点（`GROUP BY from,thread WHERE to=user AND read_at IS NULL`），修复现状 P2「徽章把已过滤的系统噪声也数进去」。

## 4.「现在发生什么」机制（填 doomscrolling gap）

核心赌注：**用一行诚实的 typing/进度行**解决等待焦虑，而不是给用户开一个仪表盘。竞品研究反复指出：bare spinner 会撒谎，15s 静默「感觉像一分钟」；**两个会动的数字（时间 + 字节/文件）**才让「它还活着吗」一眼可答。

落到三个具体组件：

**① 队长 typing 行（叙事级）**
- 队长一收到你的消息，**1–2s 内**必须出现 `队长正在 思考…` → `正在派工…` → `正在合并结果…`（短状态短语，来自 `thought_trace.status` + 首个 `AgentActivity`）。
- 这是「入流律」的最小实现，直接修现状的「绿点撒谎」根因——proof of life 永远先于 spinner。

**② 成员心跳进度行（机器级，非气泡）**

```
┌测┐ 测试成员正在 写 refund.test.ts  ⋯⋯⋯   14s · 已写 3 文件
```
- 色头像（角色色）+「正在 + 用户动词」（把 `AgentActivity.label` 翻成「写 / 跑测试 / 读文件」，**绝不**泄漏 worktree/blackboard/wake/spell）+ 三点动画 + **心跳数字**（last-output 时间 / 文件数）。
- **45s 无输出 → 降级**为 `⚠ 已 50s 无活动`（橙），不再假装在跑（用 `last_activity_at`，对应后端 wedge 检测）。这修复现状 P1「PendingBubble 在 agent 静默死亡后仍挂 60s」。

**③ 派工卡展开 = 该成员的活动时间线**
- 折叠态：一行「派给 X：做 Y · ●进行中」。
- 展开态：该成员的工具调用按时间线排列（`installing / editing X / running tests`），每行收据式（竞品研究的「折叠工具调用行」+「可滚动活动日志」）。raw CLI 流再下一层，进抽屉终端。

**断线兜底**（修复后端 P1「AgentActivity 重连后丢失 running→ok 转换」）：进度行/派工卡在 WS 重连时先打 `GET /api/agent/:id/activity` 回填，再接收流；首屏冷载同理。事实律：任何进度行只描述服务器确认的事实。

## 5. 计划 / 派工 / 交付 / 审批 / diff

| swarmx 事件 | 呈现 | 在哪 | 可操控 |
|---|---|---|---|
| **计划** | 计划卡，置顶在轴内（不是单独空间），活的 checklist `✓/◐/○`，每项标拥有它的成员 | 入流（轴顶钉住） | 派工前可批准/改：`[批准计划]`，呼应「先批计划再跑」闸门 |
| **派工** | 派工卡（左色条 + 成员色 + ●进行中），`▾` 展开成小团队树（队长→成员→依赖） | 入流（折叠态一行） | 点成员名 → 过滤轴/跳抽屉；展开看子步骤。**取代 DAG 标签** |
| **交付** | 交付卡：`N 文件 +x/−y` + 测试输出折叠块 + `[查看变更]` | 入流 | `[查看变更]` 跳抽屉变更 tab；这是「作业证据」，替掉静默 handoff |
| **审批/需要你** | 需要你卡，三要素：① 要做什么 ② 为什么 ③ 预期结果 + `[批准][拒绝][看终端]` | 入流（红/琥珀，永不消失） | 禁止裸 APPROVE/DENY；同时进收件箱 + 桌面通知 |
| **diff/变更** | 不入流。**diff 是收据，进抽屉变更 tab** | 抽屉 | 行级评论（点行→「让队长改这里」→ 带 file:line 发给成员）；合并闸：「已看全部文件」勾选才能 `[合并回 main]` |

**入流 vs 进面板的红线**（竞品研究的「叙事 / 工作面」分离 + 现状「事实律/入流律」）：
- **入流**（左侧、人能读完）：计划、派工（折叠）、交付（凝练）、需要你、失败、模型切换、CLI 回退、隔离降级。每条都服务器持久化（`kind=system` 入 messages 或 swarm_events），刷新/重连能完整重放。
- **进抽屉**（右侧、高频机器输出）：raw 终端、完整 diff、成员逐 token 流、活动时间线明细。
- **可操控的只在卡上**：批准计划、批准/拒绝、合并闸、行级评论、推进、接管/交还。

**审批与 diff 的「门」位置**（竞品研究：信任需要 agent-决定与文件-改变之间有一道闸）：swarmx worker 在隔离工作区写，门从「写前」移到「合并前」——交付卡先入流，变更 tab 审，合并闸把关，per-file accept 可挑拣。

## 6. Composer 输入区

复用现状 `MessagesPanel.tsx` 的 Composer 骨架（自增长 textarea、草稿持久化、@autocomplete、优化按钮、图片附件），重排为单卡：

```
┌─────────────────────────────────────────────────────────┐
│ 发消息给队长，或 @成员 直接对话…                          │
│ ┌opus·中┐  @  ✦优化  📎          Enter发送         ↑    │
└─────────────────────────────────────────────────────────┘
```

- **默认收件人 = 队长**；`@成员` 内联 autocomplete 直接对话某成员（复用现状 `@<role|id-prefix>` 路由）。`@` 也能选文件，给行级上下文。
- **打断 / 插话 / 排队**（竞品研究头号痛点：redirect 被静默排队，agent 跑偏几分钟）：
  - **Enter = 排队**（turn 结束后送），灰色 pending chip 显示在 Composer 上方。
  - **⌘Enter = 立即打断**（在下个工具边界 halt 当前成员并让队长重规划）。
  - 永不静默排队 course-correction——两种模式可见区分。每个会话/成员有 `Stop`。
- **模型选择**（修复现状 P2「换模型立即重启队长、无确认」）：opus/sonnet/haiku + 推理档 pill；切换前弹一句「换模型会重启队长、打断在跑的回复 [取消][确认]」，执行后入流一张「模型已切换」系统卡。
- **优化**：复用 `/api/optimizePrompt` 的 ✦ 就地改写 + undo。
- **附件**：粘贴/拖拽/路径（复用 `/api/attachment`）；上传失败自动从草稿移除该路径（修现状 P2）。
- **键盘模型**：桌面 Enter 发送 / Shift+Enter 换行；移动端 Enter 换行 + 显式发送键（复用现状）。Composer 下方常驻 `Enter 发送` 提示，三态 placeholder 加 `aria-describedby` 说明为何不可发（修现状 P2）。
- **草稿持久化**：每（会话, 收件人）keyed，挂载时恢复（复用现状）；若新会话无活队长，Composer 显式提示「发送将启动队长」而非静默 bootstrap（修现状 P2）。

## 7. 空状态 / 启动中 / 失败状态（诚实，无假绿点）

**空状态**（修现状 P1「裸『暂无消息』浪费最佳教学时机」）：对话主轴居中一句能力揭示问候 + 3 个起步 prompt（「重构这个函数 / 补测试 / 查这个 bug」），点击填入 Composer。无插画、无 4 步 tour、无 spell 推销。

**启动中**（事实律）：发首条消息后，轴里出现**启动清单卡**，逐步消费真实 bootstrap 事件：
```
◐ 准备工作区…   ✓
◐ 启动队长…     ✓
◐ 等待首次响应…  ⋯ (心跳计时)
```
- 永远不显示绿点直到**首次真实输出**才转 `● 工作中`。
- **90s 看门狗**：无消息/无活动 → 卡**原地翻转**为失败卡（不换视图、不静默）。

**失败状态**（失败卡永不消失纪律 + 完整恢复路径）：
```
┌ ✕ 队长还没法开始 ───────────────────────────┐
│ Claude Code 未登录                          │
│ ┌ claude /login ──────────── [复制] ┐       │
│ [打开终端登录] [换 Codex 重试] [重试] [看日志]│
└─────────────────────────────────────────────┘
```
- 失败会话在会话栏保持红 ✕、在收件箱保持钉住、永远可操作，直到你处理。
- **自动重试封顶 2–3 次**再翻成「需要你 / 换引擎」（竞品研究：silent fix-loop 是最坏结局；`换引擎` 是完美的有界恢复动作，复用 `RunSpellRequest` 的 CLI override + `fallback_from`）。
- **去重**：失败只在轴里一处权威呈现 + 会话栏一个 ✕ 状态。删掉现状「失败卡 + 成员栏 badge 两处重复、措辞还不一致」(P2)，且关键错误文字**绝不截断**（修现状 P1「请在终… 0:00」）。

**五态徽章**（会话栏 + 状态行统一）：`◐启动中 / ●工作中 / ⚠等你 / ✕卡住 / ✓完成`，全部服务器事实背书，绝不用「进程存活」语义。

## 8. 取舍与代价

- **牺牲了「全景可见性」**。把成员折叠进抽屉，重度并行运维者（同时盯 8 个 worker 的人）会觉得「看不够」。对他们不友好——但这是刻意的：本方向赌的是「队长心智」的多数用户，不是 Kanban 心智的少数 power user。缓解：抽屉成员板 + `⌥⇧↓ 跳下一个需要我`，但这不是默认视图。
- **牺牲了 worker 的「现场感」**。worker 不发气泡，靠队长转述+交付卡，引入一层间接。风险：**计划先行/交付凝练依赖队长 prompt 合规**（非确定性当成 UI 契约——这是现状已标的 P1 风险）。兜底:队长首条不匹配计划结构就降级为普通消息，不阻塞流；派工卡覆盖率不足时降级为文本系统卡。
- **最大风险：系统卡语法的覆盖率**。整个方向押在「派工卡+交付卡+失败卡」这套事件语法上，但 `AgentActivity` 粒度未必撑得起所有叙事（如「CLI 崩溃带 exit code」vs「CLI 永久挂起」）。这需要二期实地测试敲定卡词表，覆盖不到处一律降级为文本系统卡。
- **抽屉「跟随标签」的上下文切换成本**：用户在变更 tab 审 diff 时，若被一张新派工卡吸引回对话，抽屉状态要记忆，否则来回切伤神。靠 per-会话 localStorage（tab + 滚动位）兜，但页面 reload 仍会丢（现状已标 P2）。
