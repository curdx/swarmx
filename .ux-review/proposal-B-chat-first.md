# 方案 B · 对话优先（IM 心智）— swarmx 工作空间重设计

> 依据：.ux-review/firsthand-findings.md（真机实测）、analysis-fe-map.md（前端 IA）、analysis-be-model.md（后端能力）、竞品调研（Conductor / Crystal / Sculptor / Vibe Kanban / Codex / Cursor / Jules / Devin / Factory）与五视角批判。
> 硬约束：后端现状为基（PTY agents、blackboard、方向=worktree、spells、REST/WS，可加接口不可重写）；渐进落地；浏览器 + Tauri 双形态；正面解决 P0 状态撒谎。

## ① 一句话定位与目标心智模型

**定位一句话：**

> swarmx 是你和一支 AI 团队的 IM——每个项目一个工作空间，每件事一个频道；所有真相（包括启动失败、卡住、交付）都以消息的形式出现在聊天流里。

**第一设计公理（本方案的脊柱，直接回应 P0）：**

> **凡是没有出现在消息流里的事，对用户来说就没有发生。**
> 反向约束：agent 的任何状态变化——启动、登录失败、限流、派工、交付、退出——都必须以系统消息/卡片落进频道，禁止只活在绿点、日志或抽屉深处。失败不再需要"被发现"，它会自己说话。

这是 Slack 心智的本质优势：IM 用户天然相信"频道里没动静 = 真的没动静"，所以系统必须保证这个等式成立——这恰好就是修状态撒谎的产品化表达。

### 目标心智模型：恰好 5 个概念

| # | 概念 | 借用的隐喻 | 收编的现有概念 | 一句话定义（即 UI 内 hover 解释文案） |
|---|------|-----------|---------------|--------------------------------------|
| 1 | **工作空间** | Slack workspace | 工作空间/工作区（统一）、源码上下文（降为空间设置里的"关联代码库"） | 一个项目 + 一支 AI 团队 |
| 2 | **频道** | Slack channel | 方向/主线/主方向/thread/worktree/隔离 | 一件要做的事一个频道；`#main` 是默认频道（底层 worktree/分支只作灰字徽章） |
| 3 | **成员** | IM 联系人（带在线状态和"正在输入…"） | agent/orchestrator/调度/编排器/worker/成员 | 频道里的 AI：一位**队长**（接你的需求、拆解派工）+ 若干**队员**（干具体活，干完即走） |
| 4 | **消息** | IM 消息 + 系统事件 + 卡片 | 聊天/协作图/工作台账/录像/任务看板/通知/唤醒/派生/handoff | 你说的话、成员说的话、系统说的话（启动/失败/派工/交付都是卡片消息） |
| 5 | **变更** | PR / 代码评审 | 合并到主线/diff/清理方向 + （新增的）评审能力 | 每个频道改了什么代码；评审后合并回 main 的唯一出口 |

**概念预算纪律**：worktree、blackboard、mailbox、shim、spell、handoff、depends_on、PTY、唤醒、派生、台账、挂载——全部从用户可见文案中清除（行话防火墙，对 locale 文件加 lint）。隔离细节只在出问题时以系统消息浮现（"该频道未能隔离，正与 #main 共用目录"）。

## ③ 逐屏说明

### ③.1 入口（首次启动 / 空状态）

砍掉营销 hero、3 步引导卡、3 个功能卡（含已退役的"声明式编排"）、4 步 tour。空状态就是创建器本身：

```
┌──────────────────────────────────────────────────┐
│              开始你的第一个工作空间                  │
│                                                  │
│  项目路径  [ /Users/you/dev/my-app          ][📂] │
│            ✓ 找到 git 仓库 · 检测到 TypeScript     │
│            最近：~/dev/api-server  ~/dev/blog     │ ← 浏览器模式：recent + root-suggestions 候选
│                                                  │
│  AI 引擎   ✓ Claude Code 已登录 (claude 2.x)      │ ← 创建前预检（新探针），不再"引擎就绪"撒谎
│            ⚠ Codex 已安装但未登录 [如何登录]        │
│                                                  │
│                                  [ 创建并开始 ]   │
│  名称/颜色/关联更多代码库（默认取文件夹名）       ▸  │ ← 高级项折叠
└──────────────────────────────────────────────────┘
```

要点：
- **唯一必填项 = 项目路径**。名称默认取文件夹名，颜色自动分配（Linear 式最小必填）。
- 路径校验本就实现（350ms debounce + filesList），把结果**显式渲染在输入框下方**（实测看不见的原因要修）；Tauri 用原生选择器，浏览器给最近目录 + root-suggestions 端点候选（后端已有、UI 未用），把"粘绝对路径"降为兜底，文案改为"浏览器版暂不支持系统文件夹选择"。
- **登录预检前置到创建之前**（Conductor/Crystal 共识：依赖不满足在 setup 阶段暴露，不在任务里静默失败）。未登录不阻塞创建，但显示明确警告 + 登录指引（login_command 已在 plugins API）。
- 点「创建并开始」**不再有模态等待**：直接落进 `#main` 频道，启动过程以启动卡形式在消息流里实时打勾（见 ③.2）。假进度条（CreateWizard.tsx:822/835）整体删除。

### ③.2 创建后第一分钟：启动卡 + 计划卡（P0 的正面解）

进入 `#main` 后消息流的第一条消息就是**启动卡**，逐项消费真实 bootstrap 阶段事件（后端缺口 #1：BootstrapStage SwarmEvent + `GET /api/agent/:id/health`，把 rest.rs 既有日志事实升格为事件）：

```
┌─ ⚙ 正在组建团队 ────────────────────────────────┐
│ ✓ 启动 Claude Code              2.1s            │ ← Spawning→ShimReady
│ ✓ 登录检查通过                                   │ ← ready_plan needle 扫 "Not logged in"/限流
│ ✓ 项目引导已注入                                 │ ← "bootstrap prompt injected" 升格为事件
│ ◌ 等待队长的第一条消息… 14s（通常 40s 内）        │ ← 首响看门狗（后端缺口 #3）
│                                  [查看队长终端]  │
└─────────────────────────────────────────────────┘
```

**失败态（同一张卡原地翻转，不是另一个界面）**：

```
┌─ 🔴 队长启动失败 ───────────────────────────────┐
│ ✗ 登录检查失败：Claude Code CLI 未登录            │
│   终端原文：Not logged in · Please run /login    │ ← PTY 红字真相，自动搬进消息流
│ [打开终端登录]  [重试]  [改用 Codex 重试]         │
└─────────────────────────────────────────────────┘
```

- 看门狗规则：注入后 **90s** 无首条消息且 token 用量为 0（`/api/usage`、`last_activity_at` 现成可查）→ 卡片自动翻转为失败态，**绝不静默**。orchestrator 不再豁免于活性检测。
- [打开终端登录] 直达右侧板成员详情的 xterm，并自动进入「接管键盘」模式（见 ③.4），用户敲 `/login` 即可自救——补全"发现 → 定位 → 修复"闭环。
- [改用 Codex 重试] 消费 CLI 回退能力但**显式化**：今后任何自动回退也必须发系统消息"Claude 不可用，本频道改用 Codex [详情]"（spawn 响应加 `fallback_from`，后端缺口 #7）。

启动成功后，**队长的第一条消息固定为计划卡**（Jules/Devin"计划先行"模式；init spell 的提示词约束其先产出计划）：

```
┌─ 🧭 队长 · 计划 ────────────────────────────────┐
│ 我看完了项目（Vite + React，37 个文件）。计划：    │
│  1. □ 梳理支付回调现有逻辑（src/pay/…）           │
│  2. □ 写失败用例复现 bug                          │
│  3. □ 重构 + 跑通测试                             │
│ 验收标准：现有 12 个测试全绿 + 新增 3 个用例        │
│ [批准并开始]  [先改改]        30s 后自动开始 ▸     │
└─────────────────────────────────────────────────┘
```

- **计划卡 = 工作台账的新形态**：本体仍是 blackboard ledger markdown，但它不再是"另一个要解释的空概念视角"，而是这条消息的持久化视图——频道头 📌 常驻入口，队长每次更新台账，流里出现一条"队长更新了计划（第 2 步 ✓）"系统消息，点开看 diff。台账空状态问题（"还没写"）随之消失：没有计划卡 = 启动卡还没走完，状态自然衔接。
- 自动批准倒计时保证人走开也不卡死（可在设置改为"必须等我批准"硬门禁）。

### ③.3 主界面（频道聊天）

```
┌──┬────────────────┬──────────────────────────────────────────────┬──────────────────┐
│ ●│ swarmx-core ▾│ # 支付重构  ⎇pay-refactor  👤👤👤  📌计划 [变更 +214/−89] │ 成员 · 队员-测试   │
│ ◦│────────────────│──────────────────────────────────────────────│──────────────────│
│ ●│ ★ 需要你 (2)   │ 你：把支付回调的重试逻辑重构掉                  │ ● 正在输入…       │
│ ②│────────────────│ 🧭 队长：好，计划如下… [计划卡 📌]              │┌────────────────┐│
│  │ 频道            │ ⚙ 队长派出 队员-测试（写失败用例）[派工卡 ▸]    ││ $ npm test     ││
│ +│ # main         │ ⚙ 队长派出 队员-后端（重构回调）                ││ FAIL 2 passed 10││
│  │ # 支付重构  ←   │ 队员-测试 正在输入…                            ││ █ (实时 xterm)  ││

│  │ + 新频道        │    测试 ✓ 12 passed [查看变更] [展开测试输出]   │ 终端｜活动｜录像   │
│  │────────────────│ 🟡 队员-测试 需要你：要装 vitest@2，批准吗？     │ [接管键盘]        │
│  │                │    [批准] [拒绝] [看终端]                      │ 启动: ✓✓✓✓ 9s    │
│  │ ⚙ 设置   ⌘K   │──────────────────────────────────────────────│                  │
│  │                │ [给 #支付重构 发消息… @队长]              ➤   │                  │
└──┴────────────────┴──────────────────────────────────────────────┴──────────────────┘
 ↑空间轨(角标=未读/等你/出错)                                        ↑右侧板按需打开
```

关键设计：

- **频道头一行说完一切**，消灭五倍冗余（现状"1"同屏出现 5 次）：名称、分支灰字、成员头像堆（状态点直接画在头像上）、📌计划、变更统计。原 WorkspaceStatusStrip / 三统计 chip / "N 个 AI 在线" / "AI 引擎就绪" 全部删除。
- **成员状态 = IM 在线状态语义**（五态，每个状态由服务端事实背书，禁止进程级语义）：

| 状态 | 视觉 | 判定来源（全部已有或 P1 接线） |
|------|------|------------------------------|
| 启动中 | 灰点 + 头像呼吸 | Spawning→prompt_injected 之间（bootstrap 阶段事件） |
| 工作中 | 绿点 + 频道内"正在输入…" | AgentActivity（转录 tailer 已发 tool/system 事件）近 30s 有活动 |
| 等你 | 琥珀点 + ✉ | 发给 user 的消息未读 / needs-approval 卡未处理 |
| 卡住/失败 | 红点 | needle 命中（未登录/限流）/ shim 非 0 退出 / 看门狗超时 / orphaned-handoff 事件 |
| 已离开 | 头像变灰缩小 | Exited；队员交付后离开有告别系统消息（farewell meta 已有） |

  **"正在输入…"是本方案最 IM 化的一笔**：把转录 tailer 的工具活动翻译成 typing indicator + 一行小字"正在运行 npm test…"，agent 干活这件事第一次变得像同事在打字一样可感。
- **派工卡（收编协作图）**：队长每 spawn 一个队员，流里出现一张可折叠卡：角色、一句话任务、实时状态点、依赖关系（"等待 队员-测试 的用例"——WaitingDep 终于有了去处）。点「▸」展开成小型团队视图（树状：队长→队员们，即原 DAG 的内容），全屏协作图降级为这张卡的展开态。空状态问题消失：没派工就没有卡，无需解释。
- **交付卡（新增，竞品共识的"作业证据"）**：队员写出 handoff 时，流内发交付卡 = 改动摘要（N 文件 +x/−y）+ 测试命令输出折叠块 + [查看变更]（开右侧板 diff）。没有证据的"完成"不算完成。
- **需要你卡**：审批请求必须内嵌"它想做什么、为什么、预期结果"+ 批准/拒绝/看终端三键（不做橡皮图章式裸按钮）。同时点亮侧栏「需要你」与空间轨角标。
- **隐藏行为全部显式化为系统消息**：发消息自动拉起队长 → "队长不在线，正在启动…"（接启动卡）；切模型杀重启 → 发送前确认 + "频道模型已切换，队长正在重启"；频道改名触发隔离 → "频道已隔离到独立分支 ⎇pay-refactor，成员已在新目录重启"（P5-D 终于有了解释）；隔离失败 → 常驻琥珀系统消息"该频道未能隔离，正与 #main 共用目录，两个频道的修改会互相影响"（degraded 字段已在 API）。
- **「唤醒」概念删除**：队长/队员卡住但进程活着时，成员 hover 菜单给「提醒它检查新消息」（无确认框，幂等，点击后按钮内联变"已提醒 ✓"）；登录/崩溃类问题走错误卡的[重试]（真·重启：kill + 同 role/thread 重 spawn，后端原料齐全）。

### ③.4 右侧板：成员详情 / 变更 / 计划

Slack thread-panel 心智：点头像/卡片按钮打开，≥1536px 并排压缩聊天列（不再 overlay 遮半屏），窄屏全宽 sheet。**成员栏 `hidden 2xl:flex` 蒸发问题随常驻成员栏的取消而消解**——成员入口在频道头头像堆，任何宽度可见。

**成员详情**（替代 AgentDrawer）：
- 终端 tab：实时 xterm。**默认只读 + [接管键盘] 显式开关**（Devin "passing the keyboard"）：接管即暂停该成员防冲突；[交还] 时弹一行输入"告诉它你改了什么"，作为消息注入回会话。这是浏览器形态下用户替 agent 修登录态的正规通道。
- 启动阶段时间线（替代 SPAWN/TURN/PTY/HOOK 行话条）："启动 ✓ 2s → 登录 ✓ → 注入 ✓ → 首条消息 ✓ 9s"，数据来自 health 端点；needle 命中时此处与头像同步变红——诊断层级越深信号越诚实，绝不比浅层乐观。
- 活动 / 录像 tab：照搬现有能力（/activity 回填、asciicast 回放），「上下文」tab 死链删除（孤儿 Context.tsx 一并删）。

**变更**（新增，所有竞品都有而 swarmx 缺的那一个视角）：
- 按文件分组 diff（threads diff API 已有），行级评论 → 作为带 `文件:行号` 上下文的消息发给队长继续改（"本地的、push 之前的 PR review"）。
- 顶部动作：[从 main 更新] [合并到 main]（merge API 已有；冲突时系统消息"AI 正在解决 3 处冲突"——merge-resolver 自动 spawn 已实现，只补叙事）。合并完成 → 流内合并卡 → 提示「归档频道」。**频道至此有了生命周期终点**：建 → 干 → 审 → 合 → 归档。

### ③.5 并行工作与切换（重度用户路径）

- **开新频道 = 发新任务**（Crystal prompt-first）：侧栏 [+] 或输入框里对队长说"另开一条线做 X"→ 系统建频道、AI 自动命名、自动隔离 worktree（名字/分支都是产出不是必填项）。创建对话框只剩一个任务描述输入框 + 可选"基于已有分支"。
- **跨空间扫视**：空间轨每个头像带未读数/等你/出错角标（unreadByFrom 按 workspace_id 聚合是纯前端 join；P1 配精确未读端点替代 limit-200 近似）；侧栏「需要你」收件箱跨频道聚合，回答"我该去哪"。通知铃只对「需要你」类事件点灯并显示数量，心跳/进展永不点灯。
- **切换记忆**：per 空间记住"上次频道 + 滚动位置"（localStorage keyed by wsId），切回不再强制落 main。
- **键盘**：⌘K 收录全部 空间/频道 组合项（条目带未读/卡住徽标）；⌃1-9 直跳空间；⌥↑/↓ 频道间循环；**⌥⇧↓ 跳到下一个「需要你」**（优先队列：失败 > 等批准 > 未读）；⌘⇧N 新建空间；⌘⇧D 开变更板。
- **删除改归档**：空间行不再裸挂删除钮；空间名菜单内「归档」，确认框显示"该空间还有 N 个成员在运行，[✓] 同时终止"（默认勾选，消灭幽灵 agent 烧 token）。
- **离开也安心**：浏览器通知三类事件——成员需要你、启动失败、合并就绪（竞品视通知可靠性为产品契约）。

### ③.6 新手引导

删除 4 步 tour。替代：
- 空频道输入框上方 3 个一键示例提示词（按项目语言探测生成："总结这个项目的结构""修复 lint 报错""给 X 写测试"）——第一步行动 5 秒内可完成。
- 行话首现时 hover popover 就地解释（计划、频道、队长各一句话），不阻塞。
- 概念在用户需要时才出现：第一次并行需求出现时才提示"可以开新频道"；第一次交付时才出现变更入口。
