# swarmx 工作空间重设计（最终稿）

> **合成依据**：骨架 = 方案 C「项目-会话优先」（评委 2 得分 90、评委 3 得分 87，2/3 评委胜者，三案均分最高 86.3）。
> 嫁接自方案 A：①收件箱卡片解剖学 ②失败卡永不消失纪律 ③桌面通知契约 ④ `GET /api/board` 聚合端点 ⑤集中式诊断页 + 合并 gate。
> 嫁接自方案 B：①第一公理「不在流里=没发生」升格为硬规约（含服务端持久化前提）②「正在输入…」typing indicator ③⌃1-9 直跳 + 滚动位置记忆 ④「需要你」审批卡三要素规格 ⑤失败卡「同卡原地翻转」交互范式。
> 证据源：`.ux-review/firsthand-findings.md`（真机实测）、五视角批判、竞品调研（Conductor/Crystal/Sculptor/Vibe Kanban/Codex/Cursor/Jules/Devin/Factory）。

## 1. 诊断总结：为什么「不好用但说不出为什么」

用户说不出哪里不好，是因为**没有一个单点彻底坏掉，而是每个环节都在小额征收两种税：信任税和认知税**。单看每处都「能用」，叠加起来是弥散的不可信感与疲惫感。六条根因，每条都有实测证据：

### 根因 1：全部正向状态信号回答错误的问题（信任税之源，P0）

绿点、「1 个 AI 在线」、「AI 引擎就绪」、抽屉头部 "ready"——同屏四个绿色信号**全部同源于「进程存在」**，而用户以为它们回答「AI 能不能干活」。实测中 claude CLI 未登录时：聊天永远「暂无消息」，零错误提示、零超时兜底、零诊断入口；唯一发现真相的途径是点成员行→抽屉→肉眼读 xterm 红字 "Not logged in"（firsthand-findings P0 节）。服务端日志里其实有完整线索（bootstrap injected 等事件），UI 不消费。**信任在用户最愿意给耐心的第一分钟内破产**——而任何 bootstrap 失败（限流、CLI 崩溃、hook 卡死）表现完全一样。

### 根因 2：对象模型把基础设施抬成一等公民，概念预算超支 3 倍

新用户进门要建立约 9 个心智概念（工作空间/方向/源码上下文/agent/orchestrator/4 视角/台账/唤醒/spell），全产品 ≥25 个名词、同物最多三名（orchestrator=调度=编排器）。竞品全员把层级压成两层（项目→任务/会话），worktree/容器/PTY 全部下沉为实现细节。「方向」这个名字抽象到引导文案必须打引号自我辩护；mailbox/blackboard/wake/派生等内核行话直达用户对话框。

### 根因 3：用户真正要管理的对象（「一件要交付的事」）在体系里没有承载物，且没有终点

「方向」是长寿命的，没有「完成→审查→合并→归档」的生命周期；4 个视角（聊天/协作图/台账/录像）**没有一个展示代码 diff**——所有竞品都视为主战场的 review/merge 收口在 swarmx 完全缺位。用户派了任务，却没有地方回答「它改了什么、我怎么验收、怎么合并」。

### 根因 4：注意力架构缺失，并行即失控

跨空间未读全盲（数据在内存里却被丢弃）；「现在谁卡住了」无法 O(1) 回答（健康信号只活在 Chat 视图成员栏，且 <1536px 整体蒸发）；铃铛是布尔常亮红点；切换空间强制落回 main + 聊天 tab，零记忆；键盘没有空间级快捷键。重度用户核心循环「扫一眼→谁需要我→一键过去」每一环都断。

### 根因 5：信息预算花反了方向

同一事实同屏最多出现 5 次（"1" 出现 5 次、"main" 出现 4 次），而「agent 不能干活」0 像素；三栏 chrome 在 1536px 屏吃掉 50% 宽度；删除工作空间一次 hover 即达，幂等的「唤醒」却要过行话确认框；侧栏黄金位被三个零增量统计 chip 占据。

### 根因 6：三轮 IA 改版的历史层叠未清创

孤儿 Context.tsx 挂活的坏链接、16 个死 i18n 键、首页宣传已退役的 spell 体系、tour 描述的成员栏在窄屏根本不存在。当前 IA 是化石堆而非设计结果。

**一句话诊断**：swarmx 把内核对象模型平铺成了 UI，把「进程活着」包装成了「一切正常」，把最该花像素的时刻（agent 干不了活）留成了空白——用户每一次交互都在隐性买单，但找不到一张可以指认的账单。

## 2. 新心智模型：恰好 5 个概念 + 两条宪法

### 两条设计宪法（一切 UI 决策的上位法）

> **公理一（事实律，来自 C）**：任何 UI 状态词只许描述已被服务端事件证实的事实。绿色只在「收到首条真实输出」后出现；失败/等待是与「工作中」平级的一等状态，永不静默。
>
> **公理二（入流律，自 B 嫁接并补强）**：凡是没有出现在会话流里的事，对用户来说就没有发生。反向约束：任何状态变化——启动、登录失败、限流、CLI 回退、隔离降级、模型切换重启、改名副作用、派工、交付、合并——必须以系统卡落进会话流。**实现前提（评委 3 指出 B 自己没写的）：系统事件必须服务端持久化**（落 `messages` 表 `kind=system` 或独立 `swarm_events` 表），保证刷新/断线重连后流仍完整；禁止前端从 lossy WS 现场合成。这同时堵住「结构化事件卡覆盖率不足」开放问题的最坏情形——覆盖不足时降级为持久化的文本系统消息，但**绝不允许不入流**。

### 5 个概念（其余一切是属性或实现细节）

| # | 概念 | 一句话定义（即 UI 内 hover 解释文案） | 命名理由 | 后端载体 |
|---|------|--------------------------------------|----------|----------|
| 1 | **项目** Project | 一个代码仓库 + 它的设置（关联仓库、默认 CLI、定时任务） | 全人类预装词汇；Conductor/Crystal/VK 同名验证 | `workspaces` + `workspace_roots` |
| 2 | **会话** Session | 一件要交付的事：一条对话 + 一个隔离分支 + 一支临时 AI 小队 + 一份变更，做完即归档 | 对用过 ChatGPT/Claude 的目标用户零学习成本（评委 1 认证）；自带「有始有终」语感 | `threads` 表；slug/worktree/isolation 全部下沉为实现细节 |
| 3 | **成员** Member（队长 + 队员） | 会话里干活的 AI：一位队长拆解任务向你汇报，若干队员领活干完即退场 | 「队长」收编 orchestrator/调度/编排器三名；团队隐喻与 IM 在线状态天然兼容 | `agents`（orchestrator）+ `workers` 表 |
| 4 | **计划** Plan | 队长维护的任务计划：先给你看、可改可批，之后持续更新进度 | 「工作台账」是行话；「计划」是用户在等的那个东西 | blackboard ledger markdown，降级为会话内置顶卡 |
| 5 | **变更** Changes | 会话产出的代码 diff 与收口动作（审查→合并→归档） | 全竞品共识的主战场；现状唯一彻底缺位的概念 | 现有 `GET …/threads/:tid/diff` + `POST …/merge`（含 AI 解冲突），首次获得 UI |

**状态不是第 6 个概念**，是会话/成员的一个属性，统一为五态生命周期徽章（详见第 4 章）。

## 3. 信息架构与逐屏设计

### 3.1 路由

```
/                                  → 最近会话（无项目时 = 入口屏）
/p/:projectId                      → 项目页（会话列表；空时 = 新会话 composer）
/p/:projectId/s/:sessionId         → 会话 · 对话    (⌘1)
/p/:projectId/s/:sessionId/term    → 会话 · 终端    (⌘2)  ?member= 选成员（含「我的终端」）
/p/:projectId/s/:sessionId/changes → 会话 · 变更    (⌘3)
/inbox                             → 收件箱（跨项目聚合「等你/卡住」，A 式分诊台）
/settings(/:section)               → AI 引擎(安装+登录态) · MCP · 用量 · 归档 · 通知 · 诊断 · 偏好
（兼容：/chat/:wsId… 老路由 301 到新路径）
```

旧全局导航 8 项去向：tasks/goals → 砍（侧栏会话列表即任务列表，验收标准入计划卡）；files/terminal → 砍（人用 shell = 会话终端 tab 的「我的终端」；文件浏览由变更 tab 覆盖）；cron → 项目设置「定时消息」；mcp/usage/notifications → 设置 / 收件箱。`lib/activeWorkspace.ts` 隐式作用域联动随全局工具页消失而废除。

### 3.2 全局结构（整屏只有「一条侧栏 + 一个会话」）

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ ◆ swarmx            ⌘K                                    📥 收件箱 (⚠2) ⚙ │
├──────────────┬───────────────────────────────────────────────────────────────┤
│ swarmx-core│  修复登录重定向    ● 工作中 · 正在跑 npm test…   ⎇fix-login(灰)│
│ ▸● 修复登录…2│  队长✻ 测试员●「正在改 auth.ts…」 [对话⌘1][终端⌘2][变更⌘3 +84/−12]│
│  ⚠ 重构支付  ├───────────────────────────────────────────────────────────────┤
│  ✓ README    │ ┌ 📌 计划（队长 · 2 分钟前更新）────────────────── 折叠 ▾ ┐   │
│  ＋ 新会话    │ │ 1.✓ 复现 bug   2.✻ 修 redirect 透传   3.○ 补回归测试    │   │
│ web-app   ⚠1 │ └─────────────────────────────────────────────────────────┘   │
│ demo-app  ●2 │  队长：定位到 auth.ts:42 丢参数，方案是……                       │
│ ──────────── │  ▸ 系统卡：队长派出队员「测试员」——写失败用例（运行中 ●）        │
│ ＋ 添加项目   │  你：顺便把错误提示文案也改了                                    │
│ 归档 · 设置  │  测试员 正在输入… 「正在运行 npm test…」                         │
│              ├───────────────────────────────────────────────────────────────┤
│              │ [ 输入消息… @成员                                          ⏎ ] │
└──────────────┴───────────────────────────────────────────────────────────────┘
```

要点：
- **侧栏 = 常驻扫视仪表**（C 的全场最佳）：每行 = 五态点 + 标题 + 未读数；「等你」琥珀 ⚠ 与普通未读区分；项目折叠态仍显示聚合角标（⚠1 ●2）。深耕任何会话时余光即可巡逻，零导航税。
- **数据层（自 A 嫁接，堵 N+1）**：新增 `GET /api/board` 单次聚合端点——projects × sessions 的 `{state, unread_count, awaiting_user, last_activity_at}`（裁掉 A 最贵的 per-card diff 统计），挂载与每次 WS 重连时全量校准；增量走 WS。精确未读 = `SELECT from_agent, thread_id, COUNT(*) … WHERE to_agent='user' AND read_at IS NULL GROUP BY …`，替换 limit 200 近似。
- **「正在输入…」（自 B 嫁接）**：转录 tailer 的 AgentActivity 已有工具事件，纯前端翻译成会话头/成员 chip 上的一行小字「正在运行 npm test…」/「正在改 auth.ts…」。替代 C 原稿的 chips 脉动——重度并行时最便宜的活性信号，也是新手等待期唯一的持续安抚。
- 成员 chips 在会话头，**任何宽度可见**（成员栏 `hidden 2xl:flex` 蒸发问题随常驻成员栏的删除而消解）；三块 tab 文字只有 2 字，窄屏缩字号保留文字，不收成纯图标。
- 删除「唤醒」确认框：成员 hover 菜单「推进」，点击后按钮内联变「已提醒 ✓」（幂等、误点代价≈0）。

### 3.3 入口屏（首次使用 / 空状态）——预检拦截 + 首任务一步到位

没有营销 hero、没有 4 步 tour、没有功能卡片（含已退役的「声明式编排」宣传，全删）。第一屏就是用户要做的第一件事：

```
┌──────────────────────────────────────────────────────────┐
│                 把 AI 团队放进你的项目                     │
│                                                          │
│  项目路径  [ /Users/me/code/my-app            ] [浏览…]  │
│            ✓ 检测到 git 仓库 · pnpm 项目                  │
│            最近：~/code/my-app  ~/work/api  （候选 chip） │
│                                                          │
│  环境检查（实时，创建前就跑）：                            │
│  ✓ Claude Code CLI 已安装 (v1.2.3) · 已登录              │
│  ✕ Codex CLI 未安装  [安装指引]                          │
│                                                          │
│  你想先让 AI 做什么？                                     │
│  [ 总结这个项目的结构并给出 3 个改进建议            ⏎ ]   │
│  示例：「给 X 模块写测试」「修复 lint 报错」               │
│                                                          │
│                                  [创建项目并开始 →]       │
└──────────────────────────────────────────────────────────┘
```

**预检失败态（拦截 + 逃生口，按评委 1 嫁接语义补全）**：

```
│  环境检查：                                               │
│  ✕ Claude Code CLI 未登录                                 │
│    在终端运行：claude /login   [复制命令] [打开终端]       │
│    （login_command 来自 /api/plugins，按所选 CLI 渲染）    │
│  ── [创建项目并开始] 置灰，附原因 tooltip ──               │
│  ▸ 我知道风险，仍然创建（进入后会话将以失败卡形式提示）     │
```

- 默认**拦住**：新手从根上进不了死房间；逃生口保留给「我马上去登录」的老手，走逃生口创建的会话，启动清单第②步会如实变红（见 3.6），不会假装正常。
- 唯一必填 = 项目路径。名称默认文件夹名、颜色自动分配；CreateWizard 的两步弹窗、5 色选择器、「高级源码上下文」accordion 全部消失（关联仓库移入项目设置，接通已有 root-suggestions 端点做一键推荐）。
- 双形态：Tauri 系统文件夹选择器；浏览器复用已实现的 350ms debounce 路径校验，**修复展示位让 ✓/✕ 肉眼可见**，禁用文案改为「浏览器版暂不支持系统文件夹选择」，附最近目录候选。
- 首条任务输入框让「创建项目」与「派出第一个任务」合为一步，time-to-first-output 三案最短（评委 1 点名嫁接项）。

### 3.4 新建会话（prompt-first，⌘⇧N）——内联 composer，无模态等待

```
┌─ 新会话 · swarmx-core ────────────────────────────────┐
│ [ 修复 OAuth 回跳后丢失 redirect 参数的问题          ]   │
│ ▸ 高级（默认折叠）：基于分支 [main ▾] · CLI [Claude ▾]   │
│   · 并行开 [1▾] 份（同题多解挑赢家；N>1 时显示          │
│     预计 token 消耗 ≈ N × 单会话均值，行业甜蜜点 3-5）    │
│                                  [开始 ⏎]               │
└─────────────────────────────────────────────────────────┘
```

- 标题由首条 prompt 自动生成（≤24 字），分支 = 系统 slug，只作会话头灰字徽章——**用户不再命名和管理 worktree**。
- 提交即 navigate 进会话，对话流实时滚出启动清单（3.6）。假进度条与 60s 静默超时彻底删除。
- 创建即命名即隔离一次成型；后续改标题只改显示名（**PATCH name 与重栽解耦**——本方案唯一后端行为变更，风险见第 6 章）。
- 隔离失败落 `degraded`：会话头常驻琥珀条「隔离失败——正与主目录共享，可能与其他会话互踩 [重试隔离]」+ 同文案系统卡入流（公理二）。

### 3.5 会话 · 对话 ⌘1

见 3.2 总图。补充规格：

- **计划置顶卡** = ledger markdown 新家，带书写者健康徽章（「队长：活跃 / 已 N 分钟无活动 ⚠ / 已退出」，`last_activity_at` 现成可算）；验收标准（goals API 数据）为卡内字段。台账空状态问题消失：没有计划卡 = 启动清单还没走完，状态自然衔接。
- **队长首条消息固定为可批准计划卡**（Jules/Devin 计划先行）：分步计划 + 验收标准 + [批准并开始][先改改] + 「30s 后自动开始 ▸」倒计时（设置可改「必须等我批准」硬门禁）。**软契约兜底**（针对评委 3 对 init spell 服从性的批评）：若首条消息不符合计划卡结构，按普通消息渲染，不阻塞流程——计划卡是渐进增强而非硬依赖。
- **「需要你」审批卡三要素规格（自 B 嫁接）**：任何审批请求必须内嵌 ①它想做什么 ②为什么 ③预期结果，加 [批准][拒绝][看终端] 三键。裸 APPROVE/DENY 按钮禁止出现（「没有上下文的审查是橡皮图章」）。
- **派工系统卡**收编协作图：「队长派出队员『测试员』——写失败用例（运行中 ●）」，可展开成小型团队树；完整 DAG 收进 [⋯]→诊断视图。「已满足/等待中/派生」改译「可以开工/等上游/谁派出了谁」。
- **交付卡**（作业证据）：队员 handoff 时入流 = 改动摘要（N 文件 +x/−y）+ 测试命令输出折叠块 + [查看变更]。没有证据的「完成」不算完成。
- 输入框模型/思考强度切换：显式确认句「切换将重启队长，进行中的回复会中断」，执行后系统卡入流（公理二）。

### 3.6 启动清单与失败卡（同一张卡原地翻转，自 B 嫁接交互范式）

创建会话后对话流第一组内容（替代假进度条 + 「暂无消息」），每步消费真实服务端事件：

```
┌ 会话正在启动 ─────────────────────────────────────────┐
│ ✓ 创建隔离工作区（分支 fix-login-redirect）      2.1s │ ← ThreadChanged(ready/degraded)
│ ✓ 检查 Claude Code CLI（已安装 · 已登录）        0.8s │ ← 登录探针
│ ✻ 启动队长并交付任务……                      已等待 6s │ ← BootstrapStage 事件
│ ○ 等待队长的第一条回应（通常 40s 内）                 │ ← 首条 message/activity
└───────────────────────────────────────────────────────┘
```

任一步失败或超时（90s 首响看门狗），**同一张卡原地翻转为错误卡，不换界面、不弹窗**：

```
┌ ✕ 启动失败：Claude Code CLI 未登录 ───────────────────┐
│ 队长终端输出 "Not logged in · Please run /login"      │
│ ▸ 终端最后 10 行（点击展开，不用换页）                 │
│ [打开终端去登录]   [重试]   [换 Codex 重试]            │
└───────────────────────────────────────────────────────┘
```

- 三键布局照抄 B：[打开终端去登录] 直达该成员终端 tab 并自动进入「接管」模式；[换 Codex 重试] 显式换引擎（顺带把任何自动回退显式化：spawn 响应加 `fallback_from`，回退必发系统卡 + chip「已回退」徽章）。
- **失败卡永不消失（自 A 嫁接纪律）**：失败会话在侧栏保持红 ✕、在收件箱保持置顶，作为可 [重试]/[换引擎]/[归档] 的一等对象，直到用户处置。绝不静默回滚、绝不自动清理。
- 同一失败三处同步点亮：流内错误卡（主视图）+ 会话头徽章 ✕ + 侧栏会话行 ✕ + 收件箱角标 +1。

## 4. P0 信任修复专章

### 4.1 状态语义：五态生命周期（替换「绿点 = PTY 活着」）

| 状态 | 视觉 | 判定信号（全部服务端事实，禁止「自认」） |
|------|------|------------------------------------------|
| 启动中 | ◐ 灰 | BootstrapStage：spawned → shim_ready → mcp_ready → prompt_injected（新增事件，原料全在 rest.rs/spawn.rs 日志） |
| 工作中 | ● 绿 + 「正在 <动作>…」一行小字 | first_output/first_message 之后，且 last_activity_at 新鲜 / AgentActivity 近 30s 有事件 |
| 等你 | ⚠ 琥珀 | 发给 user 的消息未回 / 计划待批准 / 审批卡未处理 / 权限确认类 needle 命中 |
| 卡住/失败 | ✕ 红 | needle 命中（Not logged in/限流）/ shim_exit_code≠0 / 首响看门狗 90s / orphaned handoff / deps gate 300s 超时 |
| 已完成 | ✓ 蓝 | 队长宣告完成或全员 farewell 且有变更待收口；合并后 → 已合并 → 归档 |

铁律：①绿色只在收到首条真实输出后出现；②**看门狗对队长不豁免**（现状最关键 agent 恰是检测盲区）——`last_activity_at` 与 `/api/usage` token=0 两个信号今天就在 API 里，纯前端可先行；③任何 Error 必须三处同步点亮（流内卡 / 会话头徽章 / 侧栏行 + 收件箱）；④「已回退」「未隔离」等降级徽章必须配同语义系统卡入流（公理二）。

### 4.2 静默失败全枚举与 UI 兜底

| # | 静默失败场景（现状表现） | 检测信号 | UI 兜底 |
|---|---|---|---|
| 1 | CLI 未登录（绿点+「暂无消息」永远挂着） | 创建前登录探针；运行中 ready_plan needle 加 "Not logged in" 串 → AgentState::Error | 创建前：预检拦截+login_command；创建后：启动清单原地翻转错误卡三键 |
| 2 | 限流/配额（同上） | needle 加限流特征串 | 同上，错误卡注明「限流，建议 N 分钟后重试」 |
| 3 | bootstrap 卡死 / hook 挂起（永远「启动中」） | 首响看门狗：注入后 90s 无 message/activity → 广播失败事件 | 清单第④步翻红「队长没有响应」+ [看终端][重试] |

| 7 | orphaned handoff（队员永久等待，只活在服务端 warn 日志） | warn 升格为 SwarmEvent | 流内错误卡「队员 X 在等一个不会出现的信号」+ 诊断页条目 |
| 8 | deps gate 超时（等上游 300s+） | 等待计时事件 | 派工卡上「等上游已 5 分钟 ⚠」+ [让队长检查] |
| 9 | 模型/思考强度切换杀重启（隐藏行为） | 切换 API 调用点 | 发送前确认句 + 执行后系统卡「会话模型已切换，队长正在重启」 |
| 10 | 改名触发重栽、成员闪断（P5-D） | PATCH name 解耦后该路径消亡 | 解耦前过渡期：系统卡「会话已隔离到独立分支，成员已重启」 |
| 11 | 删项目留幽灵 agent 烧 token | 归档/删除路径检查活 agent | 确认框「将终止 N 个成员」默认勾选 + 诊断页幽灵清单兜底 |

| 14 | 合并冲突黑盒 | merge-resolver spawn 事件 | 系统卡「AI 正在解决 2 处冲突…」实时更新 |
