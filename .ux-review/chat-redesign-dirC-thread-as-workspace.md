# swarmx 聊天窗口重设计：「会话即编排台」

## 1. 一句话主张

**押注「边看边操控」——对话流本身就是编排界面,队长的计划、派工、交付、审批、变更全部以可操作的结构化卡内联进同一条流里,用户在不离开对话的前提下随时插话/打断/批准/合并,而不是看完再切到别的 tab 去操作。**

赌注的反面很明确:我们**不做**右侧 50% 的常驻 work surface 分屏(Devin/Manus 那套),因为 swarmx 是「1 人 + 1 队长 + N 临时成员」的编排场景,而非「1 人盯 1 个 agent 干活」的陪看场景。常驻分屏会把用户钉在「旁观者」位置;我们要的是用户作为「编排者」始终握着方向盘。终端/录像这类高频噪音 surface 退到按需抽屉,**唯独 diff review 破例进流**——因为合并是收口动作,必须在用户视线主轴上完成。

## 2. 整屏布局

设计原则:**对话流是主轴,占据中央且尽可能宽**。成员不再要一条常驻列,而是压缩成对话流顶部的一条「成员条」(横向 chip)+ 流内的派工卡承载结构;终端/录像/单成员深挖进**右侧按需抽屉**(点成员 chip 或派工卡才滑出,不抢常驻宽度)。

**关键决策**:成员从「右侧常驻列」降级为「顶部横向 chip 条」。理由——现状拆解明确指出 `成员 rail 是静态低密度,为单成员预留整列宽度`(screenshots problems)。横向 chip 在 1 个成员时只占一行高度,在 N 个成员时横向铺开仍可扫读;深挖单成员(终端/录像/活动)进**右抽屉**(复用现有 880px `AgentDrawer.tsx`)。

## 3. 消息流渲染

四种流元素,靠**左侧轨道色 + 卡片形态**区分,不靠头像堆叠:

### (a) 用户气泡
右对齐、accent-primary 实底、白字(复用现状 `MessagesPanel.tsx:1178`)。**变更**:用户消息上的「回复」按钮移除(现状 P1 问题:用户→用户回复语义为空),改为长按/hover 出「引用」——引用任意流内卡片(派工/交付/计划步骤)作为下条消息的上下文锚点。

### (b) 队长消息
左对齐,**队长是唯一拥有"完整气泡"的 agent**。左轨道用固定的队长色(如靛蓝)。这是「单一发声口」原则(rMultiAgent takeaway):队长综合后对人说话,成员通过队长说话,除非用户主动深挖。队长气泡顶部内联「▸ 思考摘要 · Ns」——**但把现状的低对比内嵌条改成可点击的独立 dim 块**(rPrinciples:reasoning 降级为 dim 折叠块,默认收起),解决现状 P2「思考摘要视觉被埋」。

### (c) worker(成员)消息——默认不进流,只进派工卡
**这是反 firehose 的核心决策**:worker 的原始 token 流、worker→队长 的握手,**默认折叠在派工卡内部**,不作为独立气泡广播(rMultiAgent:Anthropic research system / Claude Code sub-agents 的「condensed findings」原则)。

成员只在三种情况「浮」到流的顶层:
1. **交付**:成员产出结果时 → 渲染为「交付卡」(见 §5),带作业证据。
2. **求助/需输入**:成员卡住需用户拍板时 → 渲染为「审批卡 / 需要你卡」。
3. **用户主动 @ 了它**:用户 @某成员定向对话时,该成员的回复升格为独立气泡(用成员轨道色),进入一个**临时的内联子线程**(缩进、左轨道用该成员色),用完自动收拢。

### (d) 系统事件卡
居中或满宽的结构化卡,**形态明显区别于聊天气泡**(无气泡尾、有图标徽章、左侧无头像而是事件类型标签)。这就是「入流律」的物理载体——派工/交付/审批/失败/模型切换/隔离降级/引擎回退,全部是带类型的事件 chip(rMultiAgent:delegation 必须是 typed event 而非普通聊天行)。

### 多 agent 并行如何不变 firehose

```
归属与折叠的三层高度(altitude control):

  顶层(流)   = 队长发声 + 交付/审批/失败卡        ← 默认只看这个,可扫读
       ↓ 点派工卡 ▸
  中层(卡内) = 派工卡展开 → 该成员当前动作一行 +     ← 想看进展点开
                小团队树(队长→成员→依赖)
       ↓ 点[看终端] / 点成员 chip
  底层(抽屉) = 单成员 PTY/活动/录像 原始流          ← 调试才进
```

并行 5 个成员时,流里看到的是**5 张派工卡**(每张一行高度,带 `运行中 ● Ns` 活态),而不是 5 路 token 交织。这是 firehose 的物理隔离。每个成员有**稳定的角色派生名 + 固定色**(rMultiAgent:`search-pricing` 而非 `worker_7`),在成员 chip、派工卡、活态行三处复用同一色同一名。

## 4. 「现在发生什么」机制

填补 doomscrolling gap 的核心是 rAutonomous 的铁律:**agent 启动后 1–2 秒内必须冒出第一张活态信号,且是具名动作而非 spinner**。落到三个具体组件:

### 组件 A:活态行(吸附在流末,粘性)
```
………………… 现在 · 前端正在实现登录(已 23s)· 编辑 login.tsx · 3 工具调用 …………………
```
- 始终吸附在消息流最底部(`position: sticky; bottom`),滚动时不丢。
- 内容来自 `agent_activity` 流(`transcript.rs` tailer,700ms 轮询),翻译成用户动词:`正在编辑 X` / `运行测试` / `读取 Y`(rMultiAgent/rAutonomous:用户动词,绝不泄漏 worktree/blackboard/wake)。
- **诚实降级**(rCoding:two-signal status):显示「上次输出 Ns 前」,超 45s 无输出 → 变灰 +「· 已 45s 无活动 ⚠」;超 90s 看门狗 → 该成员翻转失败卡(复用现状 90s watchdog)。
- 多成员并行 → 活态行聚合:`前端实现登录(23s)· 测试写用例(12s)· 后端 ⚠ 等你`。

### 组件 B:派工卡内的活态副行
每张派工卡(运行中状态)第二行就是该成员的 live heartbeat,直接复用 `agentActivityById[agent_id]`:
```
┌─ 派工 ▸  队长 → 测试:写登录失败用例              运行中 ● 12s ┐
│   └ 测试 · 正在编辑 auth.test.ts · 上次输出 2s 前  [看终端][推进]│
└─────────────────────────────────────────────────────────────────┘
```

### 组件 C:成员 chip 的状态徽章 + 一行小字
成员条上每个 chip 复用现状五态徽章 + `正在<动作>` 一行小字(现状 `final-redesign.md:123` 已规划)。这是「最便宜的等待安抚」(rMultiAgent takeaway)。

### 关键:实时性的工程兜底
现状后端拆解的 P1 痛点——`AgentActivity WS 事件 reconnect 即丢`、`agent 状态转换不持久化`。本设计**依赖**这两个被修:活态行要在冷加载/重连后仍能回填,必须落到「入流律」要求的持久化(`agent_activity_record` 持久 ring + REST 快照 `GET /api/agent/:id/activity`)。重连时先打 `GET /api/board` 再接 WS(现状已规划)。这是落地成本里的硬依赖(见 §9)。

## 5. 计划 / 派工 / 交付 / 审批 / diff

| 事件 | 呈现位置 | 形态 | 可操控动作 |
|---|---|---|---|
| **计划** | 流顶部置顶卡(粘性) | 活态 checklist | ▸编辑步骤、approve-before-run 闸门 |
| **派工** | 入流(顶层卡) | 可展开系统卡 + 团队树 | 看终端、推进、打断该成员 |
| **交付** | 入流(顶层卡) | 作业证据卡 | 查看变更、▸测试输出 |
| **审批/需要你** | 入流(顶层卡)+ 收件箱 | 三要素卡 | 批准、拒绝、看终端 |
| **diff** | 流内就地展开(1280)/ 右 dock(1536+) | 行级评审 + 合并闸门 | 行级评论、勾选已看、合并 |

### 计划置顶卡(承载物)
解决「计划无承载物」。队长第一条消息后,计划升格为**流顶部粘性置顶卡**(复用现状台账 markdown,`final-redesign.md:188`)。每步是一个 worker 派工,状态 `✓/→/▢` 就地勾选(rAutonomous:visible 终点)。卡头显示**写卡人健康度**(`队长:活跃 / 已 N 分钟无活动 ⚠ / 已退出`)——诚实化。

**approve-before-run 闸门(关键 steering 点)**:队长发布计划后,若该会话设为「需批准」自治级,计划卡底部出现 `[批准并开工] [改计划]`,**成员不 spawn 直到用户批准**(rAutonomous/rPrinciples:Plan-to-Act gate,多 agent 跑错代价更高)。这是把「计划当合同」落地。

**软兜底**(现状 P1:首条消息是计划卡不可保证):若队长首条消息不符合计划结构 → 当普通消息渲染,置顶卡区域显示「队长尚未给出计划 · ▸ 让队长先规划」,不阻塞流。计划卡是渐进增强,非硬依赖。

### 派工卡(可展开 + 团队树)
```
┌─ 派工 ▸  队长 → 测试:写登录失败用例            运行中 ● 12s ┐   折叠态:一行
│   └ 测试 · 正在编辑 auth.test.ts · 2s 前      [看终端][推进][打断]│
└─────────────────────────────────────────────────────────────────┘

点 ▸ 展开:
┌─ 派工 ▾  队长 → 测试:写登录失败用例            运行中 ● 12s ┐
│   ┌ 团队 ────────────────────────┐                              │
│   │ 队长                          │   依赖:无                   │
│   │  └─ 测试 ● 运行中             │   产出:auth.test.ts(失败用例)│
│   │  └─ 前端 ◐ 启动中(等测试)   │   消费:计划步骤 3            │
│   └───────────────────────────────┘                              │
│   活动:✓读 auth.ts · ✓建 test 文件 · →写断言       [看终端 ›]   │
└─────────────────────────────────────────────────────────────────┘
```
替代独立 DAG tab(现状去掉「第 5 视角」)。团队树就是 DAG 的内联微缩版,`depends_on`/`produces`/`consumes` 来自现状 workers 表已有字段。

### 交付卡(作业证据)
解决「无作业证明」。成员握手 → 流里出现:
```
┌─ 交付  测试完成:新增登录失败用例    3 文件 +84/−2   ✓12通过 1失败(预期)┐
│   ▸ 测试输出(点开展开终端片段)                          [查看变更 ›]   │
└──────────────────────────────────────────────────────────────────────────┘
```
`+x/−y` 来自 worktree diff,测试输出折叠块来自该成员终端尾巴。`[查看变更]` 是 diff 入流的入口。

### diff(收口,**破例进流**)
这是本方向最重要的赌注——**review/merge 不去别的 tab,在对话主轴完成**:

- **1280px**:点交付卡的 `[查看变更]` → diff 在流里**就地展开为一张大卡**(file list + 逐文件折叠 diff),顶到当前滚动位置。用户在卡内做行级评审,卡底是合并闸门。看完收起,流继续。
- **1536px+**:同一动作 → diff 在右 dock 展开,对话不被推走,可一边看 diff 一边在 composer 指挥。

**行级评审评论**(现状 `final-redesign.md:249` + rCoding:merge-time gate):点 diff 某行 → 内联输入「让成员改这里」→ 作为带 `file:line` 上下文的消息发给该成员(回到流里成为一条定向消息)。这是 swarmx 第一个 review surface。评论「解决」用启发式(后续 diff 覆盖该行)+ 允许手动 toggle(现状 P2:启发式脆弱,留人工兜底)。

**合并闸门**(rCoding:gate 从 before-write 移到 before-merge):
```
☑ 分支有变更    ☑ rebase 检查通过    ○ 测试通过    ○ 全部评论已解决
□ 已查看全部文件(2/3)
        [ 合并回 main(条件未满足) ]   ← 勾全前置 + 看全文件才点亮
```
合并冲突 → 自动派工「合并解决」成员,**冲突解决过程作为派工卡进流直播**(入流律)。

## 7. 空状态 / 启动中 / 失败状态(诚实,无假绿点)

**事实律**:任何状态只描述服务器证实的事实;首条真实输出前绝不出现绿色。

### 空状态(教学时刻,不浪费)
替代现状裸「暂无消息」。新会话流里是 **prompt-first 创建态 + 能力揭示**:
```
┌──────────────────────────────────────────────────────────────┐
│  这个会话还没开始。告诉队长你想做什么,它会拆成计划并分派成员。│
│                                                                │
│  试试:                                                         │
│   ▸ 把登录页接到新 auth 接口,并补失败用例                      │
│   ▸ 重构 X 模块,拆成 3 个并行任务                              │
│   ▸ 给整个 components 目录补单测                                │
│                                                                │
│  引擎:✓ Claude Code 已登录 v1.x   ✕ Codex 未安装[装]          │ ← 预检,诚实
└──────────────────────────────────────────────────────────────┘
```

### 启动中(清单原地翻转)
用户首次发消息 → 流里一张**启动清单卡**,逐步消费真实 bootstrap 事件(复用现状 `final-redesign.md:195`):
```
┌─ 启动会话 ──────────────────────────────────┐
│  ✓ 创建隔离分支 (auth-fix)                   │
│  ✓ 启动队长引擎 (Claude Code)               │
│  ◐ 等待队长首次输出…(已 14s / 90s)         │ ← 真实进度,非假进度条
└──────────────────────────────────────────────┘
```
成员/队长徽章在此期是 `◐ 启动中`(bootstrap 阶段),**不是绿点**。绿色「● 工作中」只在**首次真实输出 + 新鲜活动后**才点亮(现状五态徽章)。

### 失败(原地翻转,永不消失)
任何失败(登录失败/限流/超时/孤儿等待)→ 启动清单**就地翻转**为失败卡,**不换视图、不静默**:
```
┌─ ✕ 队长还不能开始工作 ───────────────────────────────────────┐
│  原因:Claude Code 未登录。                                    │
│  在终端运行:  claude  [复制]                                  │
│  [打开终端去登录]  [换 Codex 重试]  [重试]  [▸ 查看完整日志]  │ ← 补现状缺的第 4 钮(链到诊断页)
└────────────────────────────────────────────────────────────────┘
```
- 失败会话在成员 chip 保持 `✕ 卡住` 红徽章,在收件箱保持置顶,**可操作直到用户处理**(现状「失败卡永不消失纪律」)。
- **诚实化**:成员 chip 错误文案不截断(修复现状 P1「请在终… 0:00」截断),长文案换行或链出。
- **去重**:失败只在**流内卡片**讲一次(权威),成员 chip 只显示 `✕ 卡住` 徽章 + 链回流内卡,不重复整段错误文案(修复现状 P2「失败卡和成员徽章双份错误」)。
- **限流/崩溃/超时各自定制卡**(rPrinciples:错误生命周期),不是一个通用卡。自动重试上限 2–3 次后才翻转「需要你」(rAutonomous:防静默 fix-loop)。
- **引擎回退显式化**:Claude 不可用自动回退 Codex → 流里系统卡「Claude 不可用,已切到 Codex [详情]」+ 成员 chip 「已回退」徽章(现状 `fallback_from`)。

## 8. 取舍与代价

**牺牲了什么:**

1. **牺牲「陪看单个 agent」的沉浸感**。我们不做 Devin/Manus 的常驻 50% work surface。对**想盯着一个 agent 逐字干活**的用户(尤其是不信任 agent、要全程监督的)不友好——他们得主动点「看终端」进抽屉。我们赌:编排场景里用户要的是「N 个成员的进度概览 + 收口控制」,不是「1 个成员的逐字陪看」。

2. **牺牲流的纯净度**。把派工/交付/审批/diff 全塞进一条流,流会变「重」——结构卡比聊天气泡占更多垂直空间。短会话还好,长会话(20+ 派工)流会很长。缓解:派工卡折叠成一行 + 时间分隔 + 计划卡作为粘性导航锚,但这是真实代价。

3. **牺牲并行的空间分离**。Vibe Kanban/Conductor 用 kanban 看板给每个 agent 独立 lane,空间上天然不冲突。我们坚持单流 + 派工卡,在**极高并行(8+ 成员)**时,即使折叠成一行,流里也会有 8 张卡上下堆叠,不如看板一目了然。我们赌 swarmx 典型并行度是 2–5(现状 Crystal 模型 N=3–5 甜点),单流够用;真到 8+,成员条 + 收件箱兜底,但承认这是上限。

4. **对键盘重度用户的学习成本**:QUEUE/INTERRUPT 双发送(Enter vs ⌘Enter)是已知的混淆源(rCoding:Claude Code 自己都有这个文档化困惑)。我们用「待发 chip + 插话前确认」缓解,但仍需用户建立心智模型。

**最大风险:**

- **diff 进流的性能与体验**。大 diff(50 文件)就地展开会撑爆流的滚动。**必须**虚拟化 + 默认只展开 file list,逐文件按需加载 diff。1536+ 用右 dock 缓解,1280 是真风险点。
- **「入流律」的后端持久化是硬前提**。如果 `agent_activity`/状态转换/系统事件不持久化(现状多个 P1),活态行和事件卡在重连后会空——整个「边看边操控」的可信度崩塌。这不是前端能兜的,是 §9 的硬依赖。
- **队长「单一发声口」依赖队长行为合规**。若队长不综合、直接转发成员原始输出,流照样 firehose。这是 prompt/契约层的非确定性风险(现状 P1 已标注计划卡同类问题)。

### 必须新建的后端支撑(按优先级)

1. **【P0 硬依赖】系统事件持久化 + 入流**:派工/交付/审批/失败/模型切换/引擎回退/隔离降级,必须落 `messages(kind=system)` 或 `swarm_events` 表并 flow 进流。现状这些多是 broadcast-only WS,reconnect 即丢(现状 P1×多)。这是整个方向的地基。需先做现状标注的「半天 spike:messages 表 vs swarm_events 表」选型。

2. **【P0】`agent_activity` / agent 状态转换持久化**:活态行、派工卡活动副行要在冷加载/重连后回填。现状 `AgentActivity WS 事件 reconnect 即丢`、`状态转换不持久化`(均 P1)。需持久 ring + REST 回填 + 重连先打 `GET /api/board`。

3. **【P1】结构化事件 grammar(BootstrapStage / Dispatch / Handoff / Block 事件类型)**:派工卡、交付卡、启动清单要消费具名事件。现状 tailer 的 `AgentActivity` 粒度可能不够覆盖所有叙事(现状 P2「事件卡覆盖率不确定」)——需 BootstrapStage spike,覆盖不足处降级为文本系统消息。

4. **【P1】计划卡的 approve-before-run 闸门**:需后端在「需批准」自治级下,队长发布计划后**挂起 worker spawn** 直到用户批准。现状无此 gate,需新增会话级自治级字段 + spawn 挂起逻辑。

5. **【P1】行级评审 → 定向消息**:`file:line` 上下文打包成消息发给成员。现状有 diff 端点和消息系统,需新增「评论→消息」桥 + 评论解决状态追踪(可先纯前端启发式 + 手动 toggle)。

6. **【P2】QUEUE vs INTERRUPT 路由**:`[插话now]` 需后端支持「中止在途成员 + 队长重规划」;`[发送]` 排队需「轮次结束才送达」语义。现状 send 直接投递,需新增 queued message 状态机。

**落地节奏建议**:Phase 1 只做「单流诚实化」——活态行 + 派工/交付/失败卡入流 + 计划置顶卡 + composer 双发送,**全部依赖 P0 的事件持久化 spike 先行**(现状已规划「第一天后端 spike 跑 BootstrapStage,若成本 >2× 重规划」)。diff 进流 + 右 dock + approve-gate 放 Phase 2,因为 diff 性能和 approve 后端改动更重。
