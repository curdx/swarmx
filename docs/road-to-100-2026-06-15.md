## 1. 一句话总论

> **swarmx 的核心引擎已经造好了,但「开法」是错的。** 拖分的不是缺能力,而是三件事:
> 1. 队长是被 **Stop-hook 反复续轮的交互式 CLI** 驱动,每轮还被提示词强制无脑重拉 5 个 list,trivial 任务也走完整 spawn→交付→验收的多轮链路;
> 2. **提示词(`roles/orchestrator.md`)和真实工具 API 对不上**,服务端辛苦建的 DAG / 类型化依赖 / 防环机制队长根本没用对;
> 3. 大量「可干预」原语**已经存在**(打断 / 暂停 / 杀 / PTY 注入 / @定向 / 合并闸 / 黑板审批),只是没接进主聊天流。
>
> **到 100 的路上,最高杠杆的一批改动是「改提示词 + 接线」,不是写新引擎。**

## 3. 六个战役(到 100 的路线)

### 战役 1 ｜ 队长提速:55 → 100(最高 ROI,多为纯 prompt)
根因:队长 = PTY 里的真实 `claude`,每 turn 都是一次完整 LLM 往返。`roles/orchestrator.md:162-166` 的 B1 PERCEIVE 强制每轮跑 5 次串行 list/read;trivial 还多走 spawn→交付→验收。慢不在工具(毫秒级 HTTP),在串行 LLM 往返轮数。

- **唤醒即带载荷**(M):`wake_check.rs:226` block reason 塞进新消息正文+变更 key 摘要,提示词改「载荷已给你,别再 list」。每轮砍 3-5 往返。
- **trivial self-execute 快车道**(S,纯 prompt):单文件/hello/TODO CLI 强制队长自己写、禁先跑 B1;chitchat 0 个 list 直接回。6min→亚分钟。
- **队长轮次模型分层**(M):首响/闲聊/收尾用 sonnet/低 effort,拆活/验收用 opus high。⚠️落地前用 ctx7 核实 claude CLI 会话内切 `--model/--effort` 语义。
- **list 工具加 `since` 增量**(M)。
- **prompt/prefix caching**(S):稳定前缀缓存,业界 ~85% 延迟降。
- **23s 沉默变流式进度**(S-M):每个 tool 边界 emit thought_trace step。

验证:埋 `first_token_ms`、每轮 `swarm_*` 数、trivial 端到端墙钟。目标:23s→<5s,6min→<60s。

### 战役 2 ｜ 确定性收敛:消灭 4 分钟尾巴
根因:4min ≈ worker 自测过(~30s)后迟迟不写交付 key(~3.5min LLM 黑洞) + 5s 固定 grace + 队长一整轮自由验收。`docs/handoff-protocol.md:5` 明写「done 是约定不是运行时契约」。

- **结构化 done 契约 + 服务端代跑验证命令**(M):`swarm_spawn_worker` 加 `verify_cmd`,worker 写 key 时服务端在其 cwd 实跑,据真实 exit code 判 `done` vs `.error`。
- **worker Stop-hook 加「该交付了吗」兜底**(S-M):`wake_check.rs`,turn 结束+邮箱空+key 未写 → block 提示。
- **交付即时置 plan.json step=done**(S-M):服务端代写幂等 merge,横幅不等队长那轮。
- **grace 自适应**(S):exit-0 且 verify 过的 5s→1-1.5s。

验证:埋 `worker写key→横幅`、`自测过→写key` 两段时延;故意 verify_cmd 非零,断言落 `.error` 横幅不出。

### 战役 3 ｜ 提示词↔API 对齐 + 客观验收:编排 72→100 / 正确性 90→100
最危险发现:`roles/orchestrator.md` 教队长传 `role_label`+手写 `depends_on` key;真实 schema(`tools.rs:141`)要求 `role`(slug)+类型化 `consumes` 且禁手打 key。整篇 orchestrator.md 没出现 `role=`/`consumes`/`swarm_list_roles`。后果:DAG/类型化依赖/readiness gate/防环队长根本没用对,单 worker 能跑只因不碰这些。

- **重写 orchestrator.md 对齐真实 API**(S,纯 prompt):role slug + consumes,删手打 key,加「先 list_roles」。一改让全部已建机制生效。
- **运行时 DAG 防环**(S):现成 `detect_depends_on_cycles`(`wake.rs:904`,仅静态 spell 调)接进 `spawn_worker`。
- **risk-forced verifier**(M):写代码角色标 `risk=high`,spawn 时服务端自动追加 test-runner/reviewer 消费其 done(`risk` 现为死字段,解析了从不读)。
- **客观完成信号**(M):test-runner 产出 `tests.passed`/`tests.failed`,队长按 key 类型判断,FAIL 自动触发 fixer。

验证:派全栈任务(前端 consumes 后端 API 契约),断言前端 PTY 在后端写 key 前零 token、DAG 画等待边、A↔B 互依赖被 400 拒;抓队长真实 tool_call 断言传 `role`+`consumes`(当前会失败)。

### 战役 4 ｜ 可干预护城河:88 → 100(能力已在,主要接线)
打断(`rest.rs:1940` Ctrl-C)、暂停/resume、kill、PTY 注入(`pty/lib.rs:143`)、`@定向给任意成员`(`MessagesPanel.tsx:650`)、合并闸(`workspaces.rs:839`)、黑板 HITL(`BlackboardPanel.tsx:318`)、DAG 节点动作——全已存在。

- **worker→用户 直达提问**(M,护城河增益最大、规划最薄):协议本支持(`swarm.rs:90`),角色契约故意拦在队长(`orchestrator.md:275`)。放开 + 前端渲染「带成员身份的拍板卡」,回答直接回到该 worker。
- **流内 diff 评审卡**(S→L):MergeDialog 能力做成交付时系统卡(轻量纯前端);完整版补 per-file hunk + 行级 approve/reject(spec P2-1)。
- **「打断并改」原子入口**(M):`interruptThenSend`(现仅队长)泛化到任意 @worker。

### 战役 5 ｜ 健壮性补洞:80 → 100(3 洞,2 个关乎漏钱/撒谎)

| 洞 | 现状 | 改法 | 量级 |
|---|---|---|---|
| 中途卡死无连续看门狗(红线) | 卡死进程仍 alive→UI 永久绿点 | reaper 加档:alive 但 N 分钟无活动→`stalled`(≠Error,可恢复)+「戳一下」 | M |
| server 崩溃后孤儿 CLI 不回收(漏钱) | 只改 DB UI 态,真实进程没杀 | 持久化 `shim_pid/pgid`,启动清扫 `killpg`(校验 cmdline) | M |
| 优雅关停不杀 agent | `shutdown_signal()` 只 drain HTTP | 关停遍历 registry 主动 kill | S |

外加**故障注入 CI 套件**(M)锁分:卡死、孤儿、关停三个用例当前就会红。

### 战役 6 ｜ 上手:65 → 100
- **命名统一**(S,最痛点):orchestrator 在 UI 有「队长/调度/orchestrator」三名,「方向/thread」中英混用。锁定一套用户词全量替换 i18n(~12 key,代码不动)。
- **首屏前置并行价值 + 回答「为什么不直接用 claude」**(S)。
- **开箱「试跑」并行 demo**(L):内置 sample 仓库随包 `bundle.resources`,一键建空间+开 2 方向并行,30s 看到 wow,绕过「必须有真实项目文件夹」门槛。
- **渐进式概念暴露**(M):左侧 7 路由+4 tab+首屏 15+ 概念→默认只露对话,其余 just-in-time。

# 附录:7 份子报告原文(带 file:line)

## 附录 A ｜ 响应/收敛速度(维度当前 55 分)

### 一、运行载体定性
队长**不是 API,是常驻 PTY 里的真实 `claude` 交互进程**(`--dangerously-skip-permissions`,订阅制计费):
- `cli-plugins/claude.toml`:`billing_surface = "interactive-subscription"`、`binary = "claude"`、禁 `claude -p`/SDK/ACP。
- `crates/swarmx-pty/src/lib.rs:63` `PtyBridge::spawn`:openpty + 真子进程,输出走 mpsc。
- 驱动 = 每个 turn 结束触发 Claude Stop hook → `swarmx-mcp wake-check`(`cli-plugins/claude.toml` 的 `auto_inject_stop_hook`,实现 `crates/swarmx-mcp/src/wake_check.rs`)。hook 返回 `{"decision":"block"}` 续一整轮,返回 `{}` 则 STOP 进 idle。

**这是理解一切延迟的根**:队长不是「收到消息→处理」事件循环,而是「交互式 CLI 被 PTY 注入 / 被 Stop-hook 反复续轮」驱动。每次 tool call、每次续轮都是一次完整 LLM round-trip。

### 二、队长每轮执行链路
**A. 用户发「在吗」→ 队长开口(23s 链路):**
1. 前端 `MessagesPanel.tsx:723` `api.sendMessage` → `POST /api/message` → `routes/swarm.rs:76`。**只写 mailbox,不碰 PTY**(`crates/swarmx-swarm/src/swarm.rs:360`)。
2. `routes/rest.rs:669` `drop(state.swarm.register_agent(...))` —— inbox receiver 被故意 drop,没有后台任务把 mailbox 推进 PTY。
3. 真正唤醒靠前端第二发 `MessagesPanel.tsx:737 api.wakeAgent` → `POST /api/agent/{id}/wake` → `wake::deliver_manual_wake`(`wake.rs:380`):写 `kind=wake` mailbox + **PTY 注入 `\x15<text>` → sleep 150ms → `\r`**(`wake.rs:357-366`)。
4. 队长按 `roles/orchestrator.md` B1 PERCEIVE 强制跑:`swarm_list_messages` + 读 task.ledger + 读 progress.ledger + `swarm_list_blackboard` + `swarm_list_agents`(提示词 162-166 逐条),**5 次工具=5 次串行 LLM round-trip**,再 triage、再 `swarm_send_message` 回「在」。

> 23s ≈ PTY kick(150ms) + 模型首 token 冷启动(Opus effort=high) + B1 强制 5 个 list/read 串行往返 + 最后 send_message。工具本身毫秒级(`tools.rs` 全是 reqwest 单发),**耗时全在 LLM 串行往返数**。

**B. trivial 6 分钟链路:** 队长一轮(B1 五连→triage→spawn→改 plan.json→write ledger×2→send)→ worker spawn 起**新 PTY claude 进程**(进程冷启动+MCP 注入+bootstrap,`rest.rs:1218`,含 settle 150-550ms+双 `\r` 400ms)→ worker 写文件+自测 ~30s →写 handoff →`WakeCoordinator` 收 `BlackboardChanged` → `deliver_wake` 给队长 → **队长又一整轮 B1 五连+验收+send** → worker 被 `maybe_auto_kill_on_handoff` 在 `AUTO_KILL_GRACE_MS=5000ms`(`wake.rs:72`)后回收 → 队长可能再醒一轮确认完成。

> 4 分钟拖尾 = **2-3 个额外的整轮队长 LLM turn**,每轮被 B1 强制重跑 5 个 list,轮间隔着 PTY kick + 模型冷启动。

### 三、延迟来源排序
| 排序 | 来源 | 证据 | 占比 |
|---|---|---|---|
| 1 | 队长每轮 B1 无脑重拉 5 list/read,无缓存/增量 | `orchestrator.md:162-166`;`tools.rs` 无 since/cursor | 最大且最易回收 |
| 2 | 「一切皆整轮」Stop-hook 续轮:deliver/verify/complete 各占一整轮 | `wake_check.rs` block;`wake.rs deliver_wake` | 4min 拖尾主因 |
| 3 | 模型冷启动/首 token(Opus+effort=high) | `claude.toml` effort high | 23s 大头之一 |

| 5 | `AUTO_KILL_GRACE_MS=5s` + PTY settle/双`\r` | `wake.rs:72`、`rest.rs:1413-1422` | 秒级常数 |
| 6 | 队长额外 Bash ls/cat 探目录 | Phase A SCAN 鼓励 | 叠加 |

### 四、到 100 的差距
1. 缺「增量感知」(每轮全量重拉)。2. 缺「唤醒带载荷」(`wake_check.rs:226` reason、`wake.rs:850` body 都只说「请查看」)。3. 缺「快车道」。4. 缺「模型分层」。5. 缺「实时感反馈」。6. 缺端到端计时埋点。

### 五、具体改法(收益/成本排序)
1. **唤醒即带载荷砍 B1 五连**(M):`wake_check.rs:226`+`consume_wakes`+B1 提示词。每轮省 3-5 往返。风险:reason 膨胀需截断。
2. **trivial self-execute 快车道 + chitchat 极简轮**(S,纯 prompt):triage 硬性 0 worker + 禁 B1。6min→亚分钟。风险:边界判断交 LLM,用 few-shot 收紧。
3. **队长轮次模型/effort 分层**(M):`claude.toml`+`models_config.rs`+per-turn model。⚠️先用 ctx7 查 claude CLI 会话内切语义。
4. **blackboard/messages 增量 API**(M):list 加 `since`。
5. **23s 沉默变实时进度**(S-M):thought_trace per-tool step。
6. **调参**(S):grace 5s→2s、bootstrap safety-net 缩短(保留对大 prompt 缩放,防 paste 吞 `\r` 事故)。

### 六、验证
1. first_token_ms(`deliver_manual_wake` 注入 `\r` 时刻 → `spawn.rs:401` output_rx 首字节)。目标 23s→<5s。
2. 每轮 round-trip 数(`tools.rs call_tool` 加 metric)。B1 5+→≤1。
3. trivial 端到端墙钟(隔离后端 live e2e)。6min→<60s。
4. 回归基线挂进 live-e2e。5. wake→首 token 直方图。

## 附录 B ｜ worker 生命周期收敛

### 现状:状态机(事件 vs 轮询)
happy path 几乎全事件驱动,无慢轮询:
1. **spawn**:`spawn_worker`(`rest.rs:1436`)→ `spawn_with_bookkeeping`(`rest.rs:538`)拉 PTY。服务端 mint 规范 handoff key(`rest.rs:1631-1643` `mint_handoff_key`)并写进 worker system_prompt(`build_worker_prompt`,`rest.rs:145-197`)。
2. **首轮注入(deps gate)**:`spawn_bootstrap_inject`(`rest.rs:1218`)。无 deps→等 `mcp_ready`(~1-2s)立即注入;有 deps 才走 750ms 轮询(`rest.rs:1321`,封顶 300s)。单步无 deps 不适用。
3. **运行**:codex/claude 跑工具,exit-0 自测。
4. **交付**:worker 调 `swarm_write_blackboard` 写 minted key → `swarm.rs:514` 落盘+广播 `BlackboardChanged`(事件)。
5. **队长唤醒(事件)**:`WakeCoordinator::run`(`wake.rs:465`)订阅广播→`deliver_wake`(`wake.rs:822`):写 wake 邮件 + `inject_wake_kick`(`wake.rs:289/357`)。交付后 ~150-300ms 完成。

7. **横幅**:`PlanStickyCard.tsx:46-53` allDone = `plan.json` done===total。`plan.json` 只由队长重写,故横幅严格滞后于队长验收那轮 LLM turn。
兜底:reaper 5s 扫 waitpid(`reaper.rs:32`);`handle_agent_exit` 补 `<key>.error`(`wake.rs:677`);`reconcile_after_lag`(`wake.rs:795`)。

### 4 分钟尾巴根因
**根因不在编排器/wake,而在两段无触发无契约的 LLM 回合:**
- **A(member 4min 是 2):** auto-kill 唯一触发是 worker 自己写 minted key(`wake.rs:225`)。member 4min 不掉=worker exit-0 自测后又过 ~3.5min 才真调 `swarm_write_blackboard`。纯 LLM 黑洞,**没有机制把「测试 exit 0」翻译成「立即写 handoff」**。`handoff-protocol.md:5`:「This is a convention, not a runtime contract — swarmx-core does not enforce it.」
- **B(写完 key 后到横幅):** worker 写 key→队长 150ms 内被 kick→但队长跑完整一轮 B1 PERCEIVE(`orchestrator.md:160-167`)+读产物+决策+重写 task.ledger/progress.ledger/**plan.json**+发用户消息。一整轮 Opus turn,且验收是自由「想一遍」,不读 worker 自测真实退出码。

**结论:4min ≈ (自测完→自觉写 handoff 拖延,主要) + 5s grace + (队长一整轮自由验收)。** wake 延迟可忽略(亚秒)。

### 到 100 的差距
1. 缺「自测过→立即交付」确定性桥(最大那 ~3.5min)。2. done 是约定非契约(服务端不校验/不跑验证/不附结果)。3. 验收=队长重读全世界。4. 横幅强耦合队长重写 plan.json。5. 5s grace 固定值。

## 附录 G ｜ 业界多代理编排最佳实践对标(2024–2026)

> 业界做法带一手出处;「对 swarmx 借鉴点」是基于架构的推断。

**核心判断:** 2025 共识正中两块短板——「读密集任务才适合多代理并行,写密集任务要把协调和综合收回单点」。Anthropic 多代理研究系统(综合阶段刻意收回主代理):<https://www.anthropic.com/engineering/multi-agent-research-system>;Cognition《Don't Build Multi-Agents》(警告写任务多代理脆弱):<https://cognition.ai/blog/dont-build-multi-agents>;LangChain《How and when to build multi-agent systems》(domain decomposition 比 agent 数量重要):<https://www.langchain.com/blog/how-and-when-to-build-multi-agent-systems>。**swarmx 做的是写密集真实改码,正是高危区——最值得抄的不是「派更多 worker」而是「队长协调更轻、综合更集中、上下文不每轮重拉」。**

**主题1 Planner-Executor/Orchestrator-Worker:** Anthropic lead 把计划写入 Memory 持久化(防 200k 截断),子代理拿「目标+输出格式+工具指引+边界」四件套,子代理互不知不协调(隔离换简单)。OpenAI Agents SDK 区分 agents-as-tools vs handoffs、LLM 编排 vs 代码编排(`asyncio.gather` 并行更可预测):<https://openai.github.io/openai-agents-python/multi_agent/>。LangGraph supervisor 建议直接 tool-calling 实现+删 handoff 路由消息给子代理干净 context:<https://www.langchain.com/blog/benchmarking-multi-agent-architectures>。**借鉴:派活四件套契约、确定性 DAG 调度(节点内才用 LLM)、写任务综合收回队长单点。**

**主题2 上下文与提速(最大杠杆):** Anthropic context engineering——compaction、结构化笔记/scratchpad(记忆放 context 外)、子代理只回 1-2k token 摘要、just-in-time 检索(只留轻量标识符)、"smallest set of high-signal tokens":<https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents>。延迟:prefix caching ~90% 成本降 ~85% 延迟降:<https://introl.com/blog/prompt-caching-infrastructure-llm-cost-latency-reduction-guide-2025>;并行工具调用砍最多 90%;SPAgent 动作级投机 1.65×:<https://arxiv.org/pdf/2511.20048>;流式降感知延迟。**借鉴:队长计划写盘+只喂增量、worker 回浓缩摘要、prompt caching、trivial 快路径(scale effort to complexity)。**

**主题3 验证与收敛:** Evaluator-Optimizer/Reflection(AutoGen RoundRobinGroupChat、CrewAI):generator 出稿+critic 打分反馈循环:<https://galileo.ai/blog/autogen-vs-crewai-vs-langgraph-vs-openai-agents-framework>。结构化输出做契约+自动验证+feedback loop 到 evaluator OK 才停。LLM-as-judge + ~20 条样本起步。**借鉴:队长验收前插自动 critic gate(编译/测试/diff/文件域/输出格式),不过带反馈打回;能用确定性验证就别用 LLM 验证。**

**主题4 人在环(护城河):** LangGraph `interrupt()`(暂停图+approve/edit/reject/respond,持久层可恢复):<https://docs.langchain.com/oss/python/langchain/human-in-the-loop>。Claude Code 五档权限+plan mode:<https://www.padiso.co/blog/claude-code-permission-modes-plan-default-acceptedits-bypass-production/>。Claude Code Auto Mode(两层分类器、三档权限、派活/回收双检查点、连续 3 拒/20 拦截停下交人):<https://www.anthropic.com/engineering/claude-code-auto-mode>。**借鉴:流内审批=interrupt 范式(diff 审核→批/改/打回/直接答)、plan-then-act 闸门、直接 message 某 worker、危险动作分级放行+失败熔断。**

**主题5 健壮性:** LangGraph checkpointing(节点成功存点崩了断点续)+RetryPolicy 退避,但不含自动失败检测/恢复:<https://www.langchain.com/blog/fault-tolerance-in-langgraph> / <https://www.diagrid.io/blog/checkpoints-are-not-durable-execution-why-langgraph-crewai-google-adk-and-others-fall-short-for-production-agent-workflows>。Durable Execution(Temporal,journal-replay+Saga 补偿):<https://temporal.io/blog/of-course-you-can-build-dynamic-ai-agents-with-temporal>。Anthropic:错误从出错点 resume、告诉 agent 工具失败让它自适应、rainbow deployment。**借鉴:DAG 节点 checkpoint(SQLite 在手成本低)、退避重试+区分瞬时/持久错误防级联、失败显式告知(契合诚实性红线)、git 层做 Saga 补偿、真机注入故障验证。**

**主题6 本地 CLI agent 编排(最直接对标):** **Claude Code Agent Teams(2026-02 随 Opus 4.6,几乎是 swarmx 官方镜像):** Team lead+Teammates+共享 Task list+Mailbox;teammate 独立 context 可互发消息;共享任务表自认领+依赖+文件锁防竞态;人可直接 message teammate/Escape 打断;plan approval;lead 统一 cleanup(teammate 不能自清否则状态不一致);质量闸 hooks(TeammateIdle/TaskCreated/TaskCompleted,exit 2 塞反馈);3-5 teammate 起步;**已知局限(你大概率会踩):任务状态滞后、lead 过早收工、worker 报错停住不恢复**:<https://code.claude.com/docs/en/agent-teams>。开源并行编排器多用 git worktree 隔离:<https://github.com/andyrewlee/awesome-agent-orchestrators>。**借鉴:直接拿 Agent Teams「已知局限」当回归测试清单、共享任务表文件锁、质量闸 hooks、worktree 隔离、lead 统一 cleanup 纪律。**

### 最值得抄的 5 个模式(ROI 排序)
1. **队长上下文增量化**(计划写盘+只喂增量+worker 回 1-2k 摘要+prefix caching)——治每轮重拉/token 暴涨/延迟。
2. **复杂度分级+trivial 快路径**——治 trivial 6 分钟。
3. **流内审批=interrupt 范式**——治监督只能看不能干预(护城河)。
4. **自动 critic 验收 gate**——治队长盲信/收敛拖沓。
5. **DAG checkpoint+退避重试+失败隔离**——治健壮性/级联失败。

**额外强烈建议:** 把 Claude Code Agent Teams 文档当「官方参考实现」逐条对读——设计目标最重合,连「已知局限」都帮你预演了。
