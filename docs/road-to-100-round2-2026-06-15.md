## 总判

**第二轮最重要的认知升级:把「队长是 PTY 黑盒 LLM、flockmux 只能带外推进」从「被回避的背景约束」逼成「每个方案必须正面回答的设计前提」——结果是大量看似激进的「换引擎 / 统一协议 / 服务端调度器」提案在对抗证伪下集体坍缩成同一个真理:服务端能 enforce 的最大子集 = {readiness gate 物理拦截 + MCP 写入 schema 校验 + 确定性 verify 退出码 + 死亡补 .error + 显式状态机点亮},而「内容正确 / done 名副其实 / 队长每 turn 做什么」永远只能 best-effort。**

第一轮在「提速/收敛」上做了正确的 prompt 层微调,第二轮的真增量是:
- (a) 用 ctx7/WebSearch 实测**纠正了三个会致命误导落地的事实错误**(claude-code 的 statusLine/PreCompact 上下文控制面、MCP 协议版本、6-15 计费分池);
- (b) 扎代码**挖出三个第一轮完全没看见的结构洞**(重启丢 wake 图、孤儿进程不回收只改 DB、merge 很可能合空分支);
- (c) 把第一轮散落的 prompt 微调收敛成「提示词↔API 对齐」这一条最高确定性、零风险、零依赖的真 bug。

**底线判断:第二轮没有发现任何「值得为之打破 PTY 黑盒/计费/装机/诚实性红线」的激进重写。** 所有「经证伪存活且值得做」的方案,无一例外是:在不破红线的前提下,**把已持久化的状态真正用起来(rehydrate / commit 边界 / 遥测)、把确定性决策从 prompt 下沉到 Rust(verify 退出码 / triage progress 键 / glob 相交)、把已有原语接进主流(steer / AwaitingHuman)**——而非换引擎或拿交互式 claude 当 API 用。

## 逐根问题:最终推荐

### 1. 队长驱动模型(延迟根因)
**推荐:round1 + 更优解叠加,但砍掉两条投机支柱。** moderate / 成本 L / 经证伪存活(with-mods)。

**立即做(零风险、零依赖、最高确定性):**
- **提示词↔API 对齐**(round1 战役3-A,两轮所有视角一致认定是真 bug)。`tools.rs:141-184` 的 schema 要 role slug + typed consumes 且明文「NEVER hand-type keys」,而 `orchestrator.md:226-294` 仍教 `role_label`(自由名词,不在 8 个真 slug 里)+ 手打 `depends_on` key。**队长照 prompt 走会 spawn 失败。** 纯 prompt 改、零后端、是一切下游(DAG/防环/调度)的硬前提。**应独立优先于一切支柱落地。**
- **payload-complete 唤醒,杀死 B1 五连**:`consume_wakes`(`swarm.rs:200-226`)已原子领出 ids,wake 行已带 `meta.key`(`wake.rs:857-860`),服务端零成本读黑板快照。把 `wake_check.rs:226-238` 的 reason 从「你有 N 条 wake,去 list」改成内联完整载荷。**必须同改 server endpoint + flockmux-mcp 两个二进制并保版本同步**(reason 现在 MCP 侧 `format!`,server `RETURNING` 只回 id),reason 超 STDIN_MAX_BYTES(64KB)截断退回旧文案 fail-soft。
- **接 `detect_depends_on_cycles`(`wake.rs:904`)进 `spawn_worker` 实时路径**(现仅 `run_spell` 跑,队长主用的动态派活不防环)。

**降级/缓做:**
- 服务端 triage 闸:**只先放 progress.md breadcrumb 不 block**(最安全子集,纯 Rust 可单测),每个被吞 wake 强制写一条可见 ledger 痕迹,任何不可分类一律 block(fail-loud)。
- **slash-command `model:` 分层**:ctx7 证实字段存在(haiku/sonnet/opus,override 当前会话),但 (a) codex 无此 frontmatter,claude-only;(b) interactive-subscription 态下切模型是否真生效且不破 billing 是未验证灰区。列为上线前 hard gate,过不了退回 round1 的 respawn。

**被证伪应放弃:**
- ❌ **「服务端代写 plan.json step=done」**。死因:`orchestrator.md:128` 强制队长「每次 plan 变化整文件重写 plan.json」,server 代写也走 `write_blackboard`→broadcast(`agent_id=None`),`select_targets` 不排除 system 写入→反唤醒队长→两边互相覆盖。**这恰是项目最痛的「界面撒谎」面。** 降级为「server 只读校验 + 结构化呈现,plan.json 仍由队长唯一落笔」。
- ❌ **把 `consume_wakes` 升级为「接管 dispatch 的服务端调度器」**。致命缺口(扎代码确认):plan.json 形态(`orchestrator.md:124`)只有 `{seq,task,owner_role,status}`,**没有 worker 的实质 `system_prompt`,也没有 `deps` 字段**(deps 只活在 freeform markdown `## Plan (DAG)` 行,机器读不到)。Scheduler 没有机器可读源能拿到「该给 worker 下什么 brief」→「零 Opus turn 机械推进 DAG」不成立。是对整个控制流的重写(应是 high 非 moderate),核心机制不成立,退化成 round1 子集。

### 2. 上下文与 token 经济
**推荐:用更优解替换 round1(round1 几乎空白/判死路)。** moderate / 遥测层 A=S,压缩/换代层偏 L 需真机 go/no-go / 经证伪存活(with-mods)。

**第二轮最大的认知翻盘。** round1 判定「压缩对交互式 PTY 不可行」并放弃。第二轮用 ctx7 拉 code.claude.com 发现 round1 因没查最新 CLI 文档而**完全错过的可控面**:statusLine 遥测(`context_window.used_percentage` 经 stdin 喂命令)、`PreCompact` hook、AUTOCOMPACT env、`/compact` 命令——全程不碰 billing、不靠 PTY-parse。

**立即做(A 层,纯叠加、红线安全、诚实性正收益):**
- **statusLine 遥测闭环**:`pre_spawn.rs` 写 `settings.local.json` 时加 `statusLine.command`,flockmux-mcp 加 `ctx-report` 子命令读 stdin JSON POST 回 server。把 `context_peak`(`models.rs:418`,今天只在 `usage.tsx:446` 死展示)升级成驱动决策的活信号。
- **前缀稳定化**:`wake.rs:294/850`、`rest.rs:1226` 的 `format!` 把 count/key 拼在文本里打断可缓存前缀;把可变片段挪到末尾,用 `transcript.rs:663` 的 cache_read/input 比值真机量化。

**用更省的形态做压缩/换代:**
- **压缩用 `PreCompact` hook 而非主动注入 `/compact`**:flockmux 已在注入 Stop hook,PreCompact 是同一套已验证管线,走结构化 hook stdout,**没有 paste→settle→`\r` 的 TUI 不确定性、没有与 Stop-hook/health_scanner 的时序赛跑**。它在 CLI 自己触发 compact 前注入「保留 ledger 指针/worker 句柄/未读 mailbox」。
- **换代复用现成 `reroot_thread_orchestrator`(`workspaces.rs:1426`)**,触发改为「CLI 真截断/失联」或靠遥测让队长 prompt 层自决,**不新建 server 阈值/idle 状态机**。

**被证伪应放弃:**
- ❌ flockmux 层做 token 级增量/裁剪:物理不可行,无 messages 数组句柄(`pty/lib.rs:42,143` 唯一写通道是模拟键盘)。两轮一致。
- ❌ server 主动阈值注入 `/compact`:多个事实硬伤——「只在 idle 注入」不可靠(`wake.rs:312-337` 注释明载 mid-stream 检测门已移除);阈值数字错(默认 ~83% 非 95%,`PCT_OVERRIDE` 只能调低,GH#31806);**最致命:队长正是 `claude-opus-4-8[1m]`,GH#63447 证实其 statusLine `context_window_size` 被误报成 200000,`used_percentage` 过 200k 即钳到 100%**——据此判断会误触发。
- ❌ 「AUTOCOMPACT env 零 Rust 改动」的成本谎报:`spawn_agent`(`spawn.rs:96`)签名无 per-spawn extra-env,写 settings.json env 块会被静默忽略(GH#63186)。必须真改 Rust 加 `env.insert`,且只能调低阈值的被动层。
- ❌ `DISABLE_COMPACT` 由 flockmux 全权接管:反向错误,丢掉 CLI 唯一缓存感知的压缩器。
- ❌ claude -p 起小模型压缩:6-15 后走 Agent SDK 付费池,已被 `FLOCKMUX_ALLOW_CLAUDE_PRINT` 默认禁。

### 3. done 契约与收敛/正确性
**推荐:round1 + 更优解叠加,收敛到 P0+P1 最小核心。** moderate / 成本 M-L(真成本在真机 e2e) / 经证伪存活(with-mods)。

**核心增量(四个提案高度收敛的真理):交付即验收的内联确定性门。** round1 把 verify 挂在 `maybe_auto_kill_on_handoff`(`wake.rs:582`,解散前异步旁观),FAIL 时 worker 已进 grace 无人接住→靠队长下一整轮发现。改进:验证由 server 用 `worktree.rs:40` 的 timeboxed+spawn_blocking 在 worker cwd 跑,**退出码直接决定写 done 还是 `.error`**(正确性从「LLM 判断」降为「确定性闸」),FAIL 经 MCP 工具返回值喂回**同一个仍存活的 worker**(`maybe_auto_kill` 只在写 handoff_signal 路径触发,FAIL 写 `.error` 不触发收割)→worker 同 turn 自修,零队长往返。

**立即做(P0+P1,零协议风险):**
- 加 handoff 验证字段(**改名 `handoff_verify_cmd`**,因 `verify_command`(`rest.rs:82`)已被占用 = CLI 安装探针 `claude --version`)。
- 交付门内联确定性验证。

**关键修正(揪出招牌卖点的协议错误):**
- ❌ FAIL 结果经 MCP `structuredContent` 喂回:本仓库 `flockmux-mcp` advertise 的是 **`protocolVersion="2024-11-05"`**(`handlers.rs:19`),`structuredContent`/`outputSchema` 是 2025-06-18 才进 MCP 的,2024-11-05 没有;且 `tools.rs:1-10` 有明文「text over structured」设计决策。**FAIL 反馈必须改走现有 `isError:true` + 人类可读 text**,核心机制不受损。

**被证伪应放弃:**
- ❌ 换 substrate / 建持久事件日志状态机:file+watcher 的失败模式已被三道防线全覆盖——Lagged→`reconcile_after_lag`(`wake.rs:795`)、op-log 失败→id=-1 广播(`swarm.rs:606`)、stale-key→`at>=spawned_at`(`rest.rs:1350`)。**换 substrate 解不出任何「结构上解不了的新失败模式」,两端仍是不可强制的 PTY LLM,纯成本。** 比 round1 更诚实的结论。
- ❌ 「同步 write-gate 严格优于 round1 事后旁观」的叙述:真实 chokepoint 在 `swarm.rs:514`(fs 写→op-log→`events_tx.send`→立即返回),`maybe_auto_kill` 跑在异步 broadcast loop。**应诚实承认验收只能事后异步。**
- ⚠️ **诚实性窗口(必须堵)**:落盘即 broadcast `BlackboardChanged`→唤醒队长,verify 未决期队长会把「done 已写」当完成。必须用 `Verifying` 状态 + 广播 pending_verify 标记 + `orchestrator.md` 约定「见 verifying 标记的 done 不得验收」,否则是新增界面撒谎。
- ⚠️ **执行面安全(应升 P0)**:「server 在 worker cwd 跑任意命令」是全新攻击面(现仅 git shell-out),sandbox+并发上限+时盒,把 cost 实际推向 L-XL。

### 4. 写密集并行模型
**推荐:补 round1 空白,但核心价值重排——先修真 bug。** moderate-high(服务端代 commit 改变工作树语义) / 成本 L / 经证伪存活(with-mods)。

**第二轮最重要的发现(潜在数据丢失洞):同方向 worker 没有任何 commit 边界。** `build_worker_prompt`(`rest.rs:145`)无 commit 指令,worker 在 `Shared{thread_cwd}` 裸文件系统改动;`merge_into_base`(`worktree.rs:454`)只 merge 该分支**已 commit** 的内容。**意味着 merge 很可能合一个近乎空/陈旧的分支——worker 真实改动除非 LLM 自发 commit 否则根本流不到 main(显示 Merged{files} 但实际没合 worker 的活 = 界面撒谎/数据丢失)。** round1「merge 是天然 gate」的认知只覆盖 direction→main,漏了真正会 clobber 的 worker→direction 层。

**立即做(最高价值、可独立发版):**
- **worker→direction commit 边界**:服务端代 commit(纯 git shell out,不碰 LLM)。**commit 时刻不绑 `AUTO_KILL_GRACE_MS` 定时器**(避开 LLM 仍在写文件的竞态),改由队长验收时显式触发。**必须用精确 `git add -- <owns pathspec>` 而非 `git add -A`**——后者会把用户在共享 worktree 里的未提交手改一并提交,违反 `merge_thread_handler`(`workspaces.rs:860`)「绝不动用户未提交改动」红线;owns 为空时降级 observe-only 不 commit。

**降级(数据驱动,不要过度设计):**
- **文件域调度**:直接 enforce 已存在的 `artifact_paths`(`roles.rs:79`,backend.md/frontend.md/docs-writer.md 已填 glob),**不要另造 `swarm_claim_files`/claim 表**。spawn 时同方向 glob 相交且无 depends_on→400 fail-loud + 自动转 depends_on 串行(比 round1「拒绝/警告」确定性更强、零额外 turn)。
- **阶段 0 clobber 率实测设硬 gate**:真机派 2 个写同一文件的实施 worker 量化 clobber 率。若罕见(队长基本走串行 DAG),只做轻量 enforce。

**被证伪应放弃:**
- ❌ **worktree-per-worker 默认化**:成本被严重低估——N 倍 worktree 创建(`worktree.rs:218` 各 30s 时盒)、N 次回收 merge(`merge_into_base` 只支持主 worktree,要扩两级)、每 worker reroot+重注 deps-context。仅作可选逃生舱。
- ❌ **「server 拒绝越界写」作为物理拦截**:致命——flockmux 的 MCP 工具(`tools.rs:553`)只写自管黑板,从不经手用户源文件;worker 改源码走 CLI 自带 Write/Edit 直落 OS,且带 `--dangerously-skip-permissions`(`claude.toml:14`),flockmux 只建模 stop_hook 无 PreToolUse。**源文件写路径上零拦截点。** enforce 上限只有「spawn 时 glob 相交→400」,其余靠提示词 best-effort。UI 不得把「声明了 claim」渲染成「已锁定」,否则新界面撒谎面。

### 5. 护城河:统一可干预协议
**推荐:round1 + 更优解叠加,但强制按风险分层拆发布。** moderate / 成本 L(纯加法) / 经证伪存活(with-mods)。

**核心洞察(两个提案一致):传输层不必新建,把「打断/审批/提问/纠偏」统一成一条带 `meta.subtype` 的 Intervention Envelope + 一个 `/api/steer` 多 verb 端点 + 一个 AgentState。** substrate 已就位:`messages.meta` JSON 列(migration 0012)、`in_reply_to` 线程化已贯通、MCP 分发是 `tools.rs:216` 干净 match、`SystemCard.tsx:32` 已按 subtype dispatch 且未知 subtype 优雅降级。对标 LangGraph approve/edit/reject/respond,但诚实承认「邮箱语义层非进程 checkpoint」。

**活性安全前提(本轮扎代码确认成立,round1 没核实):** blocking worker 不被误杀——`select_autokill_targets`(`wake.rs:582`)只收割「写了自己 handoff key 的 writer」,提问 worker 没写 key 故安全;FIRST_RESPONSE_WATCHDOG(`rest.rs:710`)是 ShimReady 后 90s 一次性、仅在 silent 时触发,调过 `swarm_request_user` 即有 activity 不被误杀。

**第一波(立即做,确定性正收益、零 prompt 风险):**
- 下行 `/api/steer`(redirect/respond/approve/edit):把 `interruptThenSend`(`MessagesPanel.tsx:1057`,前端临时编排且仅队长)收归后端原子端点。
- `AgentState::AwaitingHuman`(`ws_swarm.rs:115`,enum 保持 Copy,detail 走 AgentActivity)从持久 message 派生→重启可重建(解决 round1 列为关键未知的「paused 不持久化 vs 长期待审批」冲突)。
- **最小独立解先做**:`swarm.rs:80` send_message 在 to=活队长时服务端直接 `deliver_manual_wake`,消除脚本/cron/未来 API 漏 kick 的裂缝(不依赖整套协议)。

**第二波(必须真机 e2e 守门后才合):**
- 上行 `swarm_request_user` + 放开 `orchestrator.md:275` 的「别问 user 问队长」硬拦——纯 prompt 遵从性,必须隔离后端真实 claude 验证「blocking 真的 blocking、不滥用打扰」。

**被证伪/需修正:**
- ❌ 「改 wake_check 感知未答 intervention→放行 STOP」:多余。`wake_check` 是独立二进制只有 wake 计数视野,worker 发 ask 不给自己产生 kind=wake→count=0 本就返回 `{}` 自然停。**blocking 干净停是现有默认行为。**
- ⚠️ review_request 的 approve verb 触发 merge_thread 要明确与 `workspaces.rs:839`(干净树要求,脏则 409)的交互,否则「点批准却被 409 拒」是诚实性裂缝。建议先返回可执行/不可执行态,不静默触发 merge。
- ⚠️ 「堵死非 UI 发信方漏 kick」降级:cron 已 server 端 `deliver_manual_wake`(`cron.rs:242`),真实缺口仅 `/api/message` 裸端点+前端补偿。

### 6. 健壮性/确定性执行
**推荐:round1 + 补两个结构洞,但拒绝整体 Temporal 化,核心层必须降级。** moderate / 成本 M→下修(简化后接近 S) / 经证伪存活(with-mods)。

**第二轮挖出 round1 漏的两个结构洞(全部扎代码确认):**
- **洞 A:server 崩溃后真实 CLI 进程不回收(烧订阅额度)。** `mark_orphan_agents_killed`(`store.rs:2057`)只 `UPDATE agents SET killed_at`,**不发任何信号;agents 表无 pid/pgid 列**(grep 确认零结果)。崩溃后 PtyBridge 句柄全丢→真实 claude/codex 进程组继续跑。DB-only 撒谎(UI 诚实但漏钱)。修法:agents 表加 `shim_pgid`(nullable migration),启动对 orphan 行 `killpg`。**硬约束:必须 `kill(pid,0)+校验 cmdline 含 flockmux-shim/claude/codex` 防 pid 复用误杀,宁漏杀不误杀。**
- **洞 B:wake 图重启即丢。** `WakeSubs`/`ExitKeys` 是纯内存 `Arc<RwLock<HashMap>>`(`wake.rs:51/103`),`main.rs:286-289` 每次 boot 建空表,只在活 spawn 路径填。而 `workers.depends_on_json`/`handoff_signal` 是持久的。
- **洞 C:连续停滞看门狗(最低风险)。** 现有 watchdog 是一次性(`rest.rs:710`),mid-turn 卡死无人管。在 `reaper.rs` 5s 循环里加纯函数检查 `last_activity` 超阈值→发 stuck 告警(诚实可见)。零新任务、零新表。

**落地顺序:先 A(止漏钱止撒谎、不改语义),再处理 B,C 随手。**

**关键修正(对抗证伪揪出洞 B 两处硬伤,必须降级):**
- ⚠️ 洞 A 与洞 B 在主路径上语义打架:`mark_orphan_agents_killed` boot 时把**所有** live agent(含每 worker)一律标 killed,且全仓无 worker 重 spawn 路径。graceful 重启 worker PTY 已被杀光→洞 B 重建出的 wake 边对面是「DB=killed、进程已亡、永不会再写 handoff key」的死上游。
- ⚠️ wake sub 订阅者是队长 `caller_agent_id`(`rest.rs:1771`):崩溃后旧队长 id 已 killed,auto_respawn 起的是全新 id。按旧 id 重建 subs 会**唤醒死 agent**——比现状更危险的隐蔽永挂。
- **因此洞 B 降级为更简单等效解**:boot 时扫 workers 表,对 depends_on 未满足且上游已 killed 的边,**直接走 `handle_agent_exit` 合成 `.error` fail-loud 唤醒下游**(`wake.rs:677`),让 re-spawn 后的新队长靠 ledger 决定重派/收尾。**不重灌内存 wake 图**(上游进程都死了,边不会再被触发)。

**被证伪应放弃:**
- ❌ 整体 Temporal/LangGraph durable-execution 引擎:节点是不可重放 LLM + 不可回滚文件副作用,Temporal 的确定性 replay 前提根本不成立;现有机制(`handle_agent_exit`=Saga 补偿、`reconcile_after_lag`=事件重放对账、merge 拒脏树=数据安全闸)已是 checkpoint+compensation 实质,再叠一层是纯成本。两轮一致。

### 7. 计费红线(被指派质疑的约束)
**推荐:沿用 round1 的「约束不可破主干」,接受一个精确量化的有限度边角破法论证框架。** moderate / 成本 L→XL(server crate 当前零 HTTP 依赖) / 经证伪存活但 betterThanRound1=false。

**本轮真增量 = 用 6-15 实证回答「该不该破」:不该破主干。** WebSearch 证实 2026-06-15 起 claude -p/Agent SDK/GitHub Actions 走**独立信用池、按满额 API 价计费、无滚存**;交互式 PTY claude 仍走订阅。所以「切 API 做 triage/路由」在今天 = 把本免费的轻活变成花真钱,且 flockmux 看不到订阅花费(`plugins.rs:73`)→用户不知情下烧钱 = 踩诚实性红线。**这条约束今天起是真金白银的经济护城河,打破 = 反产品。**

**可做(免费混合,零计费风险):**
- 用满 **codex 的 `cli-account`(ChatGPT 订阅)**——同样不受 6-15 影响。
- 需要时起**独立常驻 haiku/sonnet 交互式 PTY triage 副手**(走订阅,非 claude -p),`spawn_with_bookkeeping`(`rest.rs:538`)是现成入口。比 round1「主队长会话内 /model 切换」更稳,但**必须进 reaper/看门狗/MAX_LIVE_AGENTS 预算**否则孤儿烧 token,且应**先把现有提示词 triage 表(`orchestrator.md:174-176`)调到位再决定是否需要副手**。

**被证伪应放弃:**
- ❌ **BYOK direct-API 旁路层**。多个证伪命中:(a) 它优化的两处 -p 特性(`rest.rs:2601/2765`)已 default-off、零成本——优化不存在的问题;(b) 经济前提错误(Agent SDK credit 按满额 API list 价计费,轻活在月度捆绑额度池内 -p 近乎订阅免费,BYOK 反而更贵);(c) **最高危泄漏面(亲自确认):`spawn.rs:298` 的 `starts_with("ANTHROPIC_")` 会转发 `ANTHROPIC_API_KEY`,只靠 `env_blocked`(`spawn.rs:451`)按 plugin `blocked_env_prefixes` 反向屏蔽,而只有 claude.toml 配了 `["ANTHROPIC_"]`,codex.toml 没配**——opt-out 弱点。代码注释声称「*_KEY 不通过」对 `ANTHROPIC_API_KEY` 是错的。若坚持 BYOK,key 必须只存 server 进程局部作用域(非 env)+ CI 硬断言。
- ❌ Managed Agents 后端:走云容器+API 价,与「本地优先+订阅额度」护城河直接冲突。仅作战略备注。

## 仍存的关键未知 / 必须真机活体 e2e 验证(静态/编译/单测一律抓不到)
