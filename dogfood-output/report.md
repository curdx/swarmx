# ① 哪块不好用（Bug / 体验缺陷）

## 🔴 H1 — 队长冷启动慢且全程无进度提示，新用户会以为卡死
- **现象**：点"发送"后，从 spawn 到队长真正开始第一回合 **约 34 秒**（日志：`12:59:03 spawn → 12:59:37 first turn`），到产出可见结果 **超过 1 分钟**。这期间聊天区只有一个空"回复"按钮，没有"正在启动队长/正在思考"的明确进度条或骨架屏。
- **真实用户反应**：第一次用的人 99% 会以为软件卡死或没反应，可能反复点击或刷新。
- **证据**：后端日志 `bootstrap: opencode started its first turn` 时间戳；UI 快照中消息区长时间只有空回复按钮。
- **建议**：发送后立刻插入一条"队长启动中…（首次启动较慢，正在拉起 CLI）"的占位消息 + 进度态；opencode TUI 队长尤其慢，应单独提示。

## 🔴 H2 — 价目表浮点精度污染，保存会写入脏数据
- **现象**：用量页价目表，单元格显示 `0.3 / 0.08 / 0.8`，但底层输入框真实值是 `0.30000001192092896`、`0.07999999821186066`、`0.800000011920929`。
- **根因**：后端 `usage.rs` 用的是 `f64`（精确），**污染发生在前端**——React number input 在解析/回填时经过了 f32 级精度丢失。一旦用户点"保存"，这些脏值就会写回 `~/.flockmux/pricing.json`。
- **证据**：`browser` 读取 spinbutton `.value` 实测；后端 `crates/flockmux-server/src/routes/usage.rs:25-31` 定义为 f64。
- **建议**：前端读写价格时统一用字符串/定点处理，或保存前 `Math.round(x*1e6)/1e6` 归一。

## 🟡 M1 — 访问不存在的路由静默跳回 /chat，没有 404 页
- **根因**：`web/src/App.tsx:128` `<Route path="*" element={<Navigate to="/chat" replace />} />`。
- **影响**：用户敲错 URL、用了失效的旧书签/分享链接时，被无声弹回首页，一头雾水，不知道是链接错了还是功能没了。
- **建议**：加一个轻量 404 页（"这个页面不存在，3 秒后回到对话"），至少给一句话解释。

## 🟡 M2 — `<html lang="en">` 写死，但默认 UI 是中文
- **根因**：`web/index.html:2` 硬编码 `lang="en"`，切换语言时不更新。
- **影响**：屏幕阅读器按英文发音读中文内容；浏览器翻译插件误判；SEO/可访问性受损。
- **建议**：i18n 切语言时同步 `document.documentElement.lang`。

## 🟡 M3 — 聊天区混入上次会话的残留任务卡片
- **现象**：发送新任务后，消息区顶部仍显示上一轮的 `dispatch opencode worker to implement missing divide and run tests` + `fixer` 卡片，和当前任务混在一起。
- **影响**：新用户分不清哪条是自己刚发的、哪条是历史，认知负担重。
- **建议**：新任务与历史任务用明显分隔（时间分割线/"本次会话"分组），或折叠历史。

## 🟡 M4 — 全站整页刷新式导航，丢失 SPA 快速切换体验
- **现象**：点侧栏每个一级页面（文件/终端/MCP…）都触发 Vite 重连（`[vite] connecting...` 每页一次），即整页重载而非 SPA 软导航。
- **影响**：页面间切换有白屏/重载感，比纯前端路由慢，状态丢失。
- **注**：这点需在生产构建（非 dev）下复测确认，dev 的 HMR 重连不完全等于生产行为。
- **建议**：确认一级导航走 React Router 客户端跳转而非 `<a href>` 整页跳。

## 🟡 M5 — MCP 页缺 server 名称/标题，两组开关无上下文
- **现象**：MCP 页有两组 Claude/Codex 开关，但第一组没有清晰的 server 名称标题，只孤零零一个"文档"链接，用户不知道这组开关管的是哪个 MCP server。
- **建议**：每组开关上方加 server 名称 + 一句话说明。

## 🟡 M6 — 默认队长 claude 静默降级到 opencode，用户无感知
- **现象**：日志 `captain_cli resolved spell=init from=claude to=opencode`，说明默认想用 claude 当队长，但实际 fall back 成了 opencode。`model config loaded clis=2`（只加载 2 个 CLI 的模型配置，而 UI 说 4 引擎可用）。
- **影响**：用户以为在用 claude，实际用了别的引擎，行为/成本/质量都不同却毫无提示。
- **建议**：降级时在 UI 明示"队长已从 Claude 切换为 OpenCode（原因：…）"。

# ② 哪块可以升级（增强）

## 🟢 U1 — 给"队长在思考"做实时工具流的可视化
台账里能看到队长调了 `swarm_read_blackboard / glob / bash / read / apply_patch` 一长串，这其实是很棒的可观测性素材。建议在聊天区就做成实时滚动的"工具调用流"（类似 Claude 的工具卡片），让用户直观看到队长在干嘛，顺带缓解 H1 的"像卡死"问题。

## 🟢 U2 — 空状态引导可以更"会带人"
任务看板"0 个任务"、用量"暂无用量数据"等空状态目前只有一句话。可加一个"试试这个"的引导按钮直接拉起第一个动作，降低新用户冷启动门槛。新手引导（1/4 步）是个好开始，但只覆盖了工作空间概念。

## 🟢 U3 — 价目表批量操作
价目表是逐格编辑。可加"从剪贴板粘贴/导入 JSON""一键同步 LiteLLM 最新价格"等批量手段。

## 🟢 U4 — 首屏给"快速上手"路径
新用户落地在一个带历史数据的 chat，但"该先干什么"不够突出。"把要做的事交给队长"+3 个示例任务是对的方向，可以再往前一步：把"新建工作空间→挂源码→发任务"做成一条高亮的首次引导路径。

# ③ 你可能没想到的（隐患 / 盲点）

## ⚠️ R1 — "浏览整个文件系统"开关是高权限操作，缺二次确认
文件页有复选框「解除工作区隔离，可浏览整台机器的文件」。文案有警告，但**没有二次确认**。一旦勾上，这个本地 Web 应用就能浏览整机文件——如果未来端口暴露（哪怕误配）或被 CSRF/恶意页面利用，风险面陡增。建议：勾选时弹确认框，且会话级而非持久记忆该状态。

## ⚠️ R2 — 单实例锁是好设计，但 stop 不彻底会留孤儿
今天就实测到：我新起后端被既有实例的 `server.lock` 干净挡回（很棒）。但 `test-stack.sh stop` 依赖 PID 文件 + `lsof` 兜底；后端日志也出现过 `settled orphan live agents orphans=1`——说明非正常退出会残留"孤儿 agent"。建议确认 reaper 对孤儿 agent / 孤儿 PTY 子进程（真实 claude/codex 进程）的回收在各种崩溃路径下都可靠，否则可能泄漏真实 CLI 进程在后台烧 token。

## ⚠️ R3 — 真实 token 消耗无"刹车"，发一句话可能跑很久
我那条简单任务，opencode 队长思考了 1 分多钟、调了近 10 个工具。**真实用户发一个含糊的大任务，队长可能自顾自跑很久、烧掉大量订阅额度**，而 UI 上的成本要等会话日志采集后才显示（用量页注明"PTY 模式下 CLI 不回传花费"，是事后估算）。建议：① 发送前对"大任务"给预算提示；② 实时显示累计 token/预估成本；③ 目标页已有"Token 预算"字段，可把它变成硬性熔断而非仅记录。

## ⚠️ R4 — `~/.flockmux/` 下散落 per-agent 配置文件，缺清理
日志显示每个 agent 会写 `~/.flockmux/opencode/opencode-834a8ac6.json` 这类隔离配置。长期高频使用后，这些 per-agent 文件是否有 GC？建议确认有清理策略，否则目录会无限膨胀。

## ⚠️ R5 — 历史数据迁移/版本兼容
隔离栈的 `/tmp` 数据目录里有上次跑留下的 workspace，这次能正常加载——好。但 SQLite schema 升级（`sqlx::migrate!`）在真实用户跨版本升级时是否平滑？建议把"旧版本数据 → 新版本"列入发版校验，避免用户升级后历史 workspace 打不开。

## 测试范围与边界说明

**测了**：9 个一级页面 + 4 个工作空间内 tab（聊天/协作图/台账/录像）全部导航打开并抓 console；核心流程端到端（发任务→真实改盘）；输入框按钮启用逻辑；价目表精度；路由 404 行为；语言属性。

**没深测（建议后续补）**：
- 终端"连接终端"的真实 PTY 交互
- 多 agent 并发 swarm（这次只跑了单队长）
- 录像/回放播放器的实际播放
- 表单的极端边界（超长输入、特殊字符、空提交各页）
- 生产构建（`tauri build` 安装版）下的行为——按项目头号原则，这才是真实用户路径，**强烈建议在装包版上重跑本报告的核心流程**
- 响应式/窄屏布局

**测试环境**：macOS arm64（Mac mini）· cargo/rustc 1.95.0 · node v24.15.0 · claude/codex/opencode/reasonix 均在 PATH。
